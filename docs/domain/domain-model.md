# 코너학습 운영 시스템 — 유비쿼터스 언어 및 엔티티 행동 정의

> 기술 스택/DB/API 설계는 다루지 않음. 도메인 모델(용어·상태·규칙)에 집중.
> 전제: 조 20개, 코너 10개, 트랙(관리자가 생성/삭제하는 독립 단위, 진행자와 1:1)은 코너별로 1개 이상, 관리자 2명(완전 동등 권한) 이내.
> 플랫폼: 진행자/관리자 클라이언트는 **네이티브 앱**(QR/MFC 스캔, 화면 상시 켜짐 유지 목적). 조장은 여전히 앱 설치 없이 QR 배지만 소지.

---

## 1. 유비쿼터스 언어 (Glossary)

| 용어 | 정의 |
|---|---|
| **캠프 (Camp)** | 코너학습이 운영되는 행사 1회차를 식별하는 최상위 엔티티. 하위 모든 엔티티(조·코너·트랙·방문)는 캠프 ID에 귀속된다. |
| **조 (Group)** | 참가 학생들로 구성된 이동 단위. 조장 1명이 대표로 QR 배지를 소지. 배지 등록을 통해서만 생성된다. |
| **조장 (Group Leader)** | 조를 대표하는 QR 배지 소지자. 스캔 행위의 대상(피동적 존재). |
| **코너 (Corner)** | 고정된 장소에서 운영되는 학습 부스. 목표 소요시간(기본 10분)을 가짐. |
| **트랙 (Track)** | 관리자가 생성·삭제하는 독립 처리 단위. 특정 코너에 귀속되며, 이 귀속은 불변이다. |
| **진행자 (Facilitator)** | 특정 트랙에 배정되어 조의 입장을 처리하고 퇴장을 확인하는 스태프(팀 단위 배치). |
| **관리자 (Admin)** | 전체 코너·조 현황을 조회하고 규칙을 실시간 조정할 권한을 가진 최상위 역할. 최대 2명. |
| **방문 (Visit)** | "특정 조가 특정 코너(트랙)에서 수행한 1회의 활동" 단위. |
| **방문 로그 (Visit Record)** | 방문의 시작시각·종료시각·소요시간·목표시간편차를 담은 기록. 조·코너 조합당 유일해야 함. |
| **순회표 (Group Itinerary / Progress Map)** | 한 조가 10개 코너 중 무엇을 완료했고 무엇이 남았는지 나타내는 상태 집합. |
| **완주 (Completion)** | 한 조가 10개 코너 전부에서 방문을 "완료" 상태로 종료한 시점. |
| **대기열 (Queue)** | 코너 앞에서 순서를 기다리는 조들의 물리적 줄(시스템 관리 범위 밖). |
| **유휴 상태 (Idle State)** | 코너(모든 트랙)에 진행 중인 조가 전혀 없는 상태. |
| **병목 지점 (Bottleneck)** | 목표시간편차가 지속적으로 양(+)인 코너. 판정 기준은 비율과 최소 표본에 의한다. |
| **목표 시간 편차 (Deviation)** | `실제 소요시간 - 목표시간`. 코너 단위 부하 판단 지표. |
| **규칙 변경 (Rule Override)** | 관리자가 특정 코너의 목표시간/트랙 운영 방식/조 배정 로직을 실시간으로 변경하는 행위. |
| **인가 세션 (Authorized Session)** | 진행자/관리자가 로그인 후 발급받는, 본인 역할 범위 내에서만 조작을 허용하는 세션. |
| **QR 배지 (Badge)** | 조와 무관하게 미리 대량 인쇄해두는 물리적 QR 식별 단위. 캠프가 끝나도 재사용 가능하다. |
| **배지 등록 (Badge Registration)** | 미배정 QR 배지를 이번 캠프의 특정 조 이름과 묶어 조(Group) 엔티티를 생성하는 행위. |
| **조 식별 배지 (Group QR Badge)** | 배지 등록이 완료돼 특정 캠프의 특정 조에 묶인 QR 배지. 위변조·복제 방지 대상. |
| **수동 처리 (Manual Check-in)** | QR 배지가 스캔 불가능할 때, 진행자가 앱에서 조를 선택해 입장을 기록하는 대체 입력 수단. |
| **트랙 PIN (Track PIN)** | 진행자 인증에 쓰이는 6자리 숫자 코드. 트랙 생성 시 시스템이 자동 발급하며 트랙 생명주기에 귀속된다. |
| **PIN 카드 (PIN Card)** | 트랙 PIN을 인쇄해 현장에 비치하는 실물 카드. |
| **트랙 교체 (Track Replacement)** | "코너 담당 변경"에 대응하는 관리자 조작. 기존 트랙 삭제와 신규 트랙 생성을 원자적으로 수행한다. |
| **기기 등록 요청 (Device Registration Request)** | 진행자 기기가 등록 코드로 신청 시 생성되는 대기(PENDING) 상태의 요청. |
| **신뢰 기기 (Trusted Device)** | 관리자 승인을 거쳐 등록된 기기. 트랙 PIN 화면에 진입할 수 있는 자격을 가진다. |
| **공지 채널 (Broadcast Channel)** | 관리자가 전체 ACTIVE 트랙에 단방향으로 공지를 보내는 채널. |
| **트랙 다이렉트 (Track Direct Thread)** | 관리자와 특정 트랙 사이의 1:1 양방향 대화 스레드. |

## 2. 엔티티 정의 및 속성

### 2.0 캠프 (Camp)
- **속성**: 캠프 ID, 이름, 기간(시작~종료 일시), **상태**(PENDING/ACTIVE/ENDED), 병목 판정 파라미터(minSamples, bottleneckRatio)
- **역할 및 책임**: 모든 데이터의 최상위 경계(Scope). 한 캠프 내에서 코너학습 프로그램은 1회만 운영된다.
- **상태 정의 및 접근성 경계**:
  - **PENDING (준비 중)**: 캠프 및 코너, 트랙, 조의 사전 설정 단계. 진행자 로그인은 불가능하다.
  - **ACTIVE (진행 중)**: 실시간 운영 단계. 대시보드, 조 현황, 기기 승인, 메시지 등 모든 기능이 활성화된다.
  - **ENDED (종료됨)**: 리포트 조회만 허용되는 사후 분석 단계. 실시간 운영 데이터 및 설정 수정은 불가능하다.
- **상태 전이**: 오직 관리자의 명시적 조작으로만 PENDING → ACTIVE → ENDED 단방향 전이가 발생하며 역행은 불가능하다. (구체적인 전이 시나리오는 `scenarios.md` Feature 2-g, 2-i 참고)

### 2.1 조 (Group)
- **속성**: 조 ID, 소속 캠프 ID, 조 이름/번호, 순회표(10개 코너 각각의 상태), 현재 상태, 현재 위치(마지막으로 스캔된 코너), 완주 여부, 배정된 배지 ID
- **역할 및 책임**: 코너학습을 수행하며 위치 및 진행률 상태를 관리하는 기본 엔티티.
- **도메인 규칙 (불변식)**:
  - 한 조는 동시에 하나의 코너(트랙)에서만 `IN_PROGRESS`일 수 있다.
  - 한 조는 동일 코너를 중복해서 완료(`COMPLETED`)할 수 없다(중복 방문 금지).
  - 완주 여부는 10개 코너 모두 `COMPLETED`된 상태이나, 프로그램 종료 시 미완료 코너가 있어도 시스템 오류로 보지 않고 부분 완주로 처리한다.

#### 2.1-a QR 배지 바인딩 및 라이프사이클
- **물리적 자산으로서의 배지**: 배지(Badge)는 캠프와 무관하게 존재하며, 캠프 시작 전 대량 생성 가능하다.
- **배지와 조의 매핑**: 미배정(UNASSIGNED) 배지가 등록 단계를 통해 특정 캠프의 조에 바인딩되어 배정됨(ASSIGNED) 상태가 되며, 이 등록 행위만이 조 엔티티를 생성하는 유일한 경로이다.
- **자산 재사용**: 캠프 종료 후 배정되었던 배지는 다시 미배정 상태가 되어 새 캠프의 새 조에 등록될 수 있으나, 과거 조와 방문 기록은 배지 재사용과 무관하게 영구 보존된다.

### 2.2 코너 (Corner) / 트랙 (Track)
- **코너 속성**: 코너 ID, 소속 캠프 ID, 이름, 목표시간(기본값 10분, 트랙별 override 가능), 현재 ACTIVE 트랙 목록, 전체 상태(미가동/정상/유휴)
- **트랙 속성**: 트랙 ID, `corner_id`(불변), 트랙 PIN, 상태(ACTIVE/DELETED), 배정된 진행자 세션, 현재 처리 중인 조
- **역할 및 책임**:
  - 코너는 조가 방문하여 학습하는 대상 장소이며, 목표 소요시간을 설정 및 제공한다.
  - 트랙은 코너의 동시 처리 용량을 규정하는 독립 처리 단위이다. 진행자 기기와 1:1로 인증 및 세션이 매핑된다.
- **도메인 규칙 및 삭제 경계**:
  - 트랙당 동시에 `IN_PROGRESS` 상태인 방문은 최대 1개이다.
  - `IN_PROGRESS`인 방문이 있는 트랙은 삭제가 하드 블록(Hard Block)된다.
  - 트랙 삭제로 코너의 ACTIVE 트랙이 0개가 되는 경우 경고 및 확인 단계를 거쳐 허용된다(미가동 상태 정상 허용).
  - 트랙 삭제 시 발급된 PIN은 무효화되고, 해당 트랙의 로그인 세션도 즉시 무효화된다.

### 2.3 트랙 교체 (Track Replacement)
- **정의**: 기존 트랙 삭제와 신규 코너로의 신규 트랙 생성을 원자적으로 묶어 처리하는 관리자 오퍼레이션.
- **제약 및 경계**: 트랙 삭제의 하드 블록 제약이 동일하게 적용된다. 교체 시 기존 세션의 연속성이 유지되며, 기로그인된 신뢰 기기는 새 PIN 입력 없이 새 세션으로 전환된다. (구체적 시나리오는 `scenarios.md` Feature 2-c 참고)

### 2.4 진행자 (Facilitator) 및 기기 인증
- **역할 및 책임**: 특정 트랙에 배치되어 조의 입장(스캔/수동)과 퇴장(확인)을 기록하고 운영한다.
- **인증 모델 (트랙 PIN)**: 진행자는 개인 계정이 없으며, 트랙 생성 시 발급된 6자리 숫자 PIN으로 로그인하여 해당 트랙 스코프의 세션을 획득한다.
- **세션 생명주기 경계**:
  - 유휴 타임아웃(Idle Timeout)으로 세션이 만료되지 않는다.
  - 세션 종료는 오직 관리자의 강제 로그아웃, 트랙 삭제, 캠프 종료, 트랙 PIN 재발급의 4가지 조건에 의해서만 발생하며, 이는 방문 처리 중(BUSY)이더라도 예외 없이 즉시 실행되어 방문을 미완료로 중단한다.

#### 2.4-b 기기 신뢰 모델 (Device Trust)
- **신뢰 장치 게이트**: 신뢰 토큰(서버 발급)이 없는 미등록 기기는 PIN 입력 화면에 도달할 수 없다.
- **토큰 상태 관리**: 토큰은 `PENDING` -> `APPROVED` 또는 `REJECTED`로 상태가 관리되며, 분실 시 `REVOKED`로 신뢰를 철회할 수 있다.
- **지연 정책**: 이미 승인된 신뢰 기기에서 PIN 입력이 연속 실패하는 경우에만 점증형 지연(스로틀링) 정책이 적용된다. (구체적 시나리오는 `scenarios.md` Feature 3-b 참고)

#### 2.5 관리자 (Admin)
- **속성**: 관리자 ID, 비밀번호, 세션 상태
- **역할 및 책임**: 캠프 전반의 생성, 종료, 코너/트랙 설계, 기기 승인, 규칙 실시간 변경(목표시간 등), 공지 발송 및 트랙 다이렉트 소통을 담당한다.
- **권한 경계**: 관리자 2명은 완전히 동등한 최고 권한을 가진다. 상충하는 실시간 규칙 변경 시 최종 쓰기 반영(Last Write Wins) 정책을 따른다.
- **PIN 재발급 (Regenerate PIN)**: 트랙 교체와 달리 트랙 엔티티(ID 및 이력)를 유지한 상태에서 PIN 값만 갱신하며, 기존 로그인된 진행자 세션을 즉시 무효화한다. (구체적 시나리오는 `scenarios.md` Feature 2-j 참고)

### 2.6 방문 (Visit)
- **속성**: 방문 ID, 조 ID, 코너 ID, 트랙 ID, 시작시각, 종료시각, 상태(진행중/완료), 소요시간(종료시각-시작시각), 목표시간편차(소요시간-목표시간), 입력 방식(input_method: `QR_SCAN`/`MANUAL`)
- **역할 및 책임**: 특정 조가 특정 코너에서 수행한 활동의 시작과 끝, 그리고 시간 지표를 기록한다.
- **입력 방식 경계**:
  - **시작 (입장)**: QR 배지 스캔(`QR_SCAN`) 또는 QR 훼손 시 진행자가 조를 선택해 입력하는 수동 처리(`MANUAL`) 방식이 적용된다.
  - **종료 (퇴장)**: 한 트랙당 동시 진행이 1개로 제한되므로 재식별(스캔) 없이 화면 확인(2회 탭 확인)만으로 종료한다. 퇴장은 별도의 입력 방식을 구별하지 않는다.
- **대기열 격리**: 대기 상태(`QUEUED`)는 도메인 모델에 존재하지 않으며, 시스템은 대기 순서나 대기 중인 조를 데이터로 보관하지 않는다.

#### 2.6-b 병목 판정 (Bottleneck Calculation)
- **기준**: 절대 편차 시간이 아닌 캠프 단위로 지정된 비율(`bottleneckRatio`) 및 최소 완료 표본(`minSamples`) 파라미터를 기반으로 실시간 집계 판정한다.
- **수식**: `(평균 실제 소요시간 - 목표시간) / 목표시간 >= bottleneckRatio` (단, 완료 표본 수 >= `minSamples`)
- **경계**: 대시보드의 병목 여부 이진 노출에만 사용되며, 병목 순위 랭킹(Report)과는 분리된다.

### 2.9 메시지 (Message)
- **역할 및 책임**: 관리자와 진행자 간 실시간 조율을 위해 제공되는 단방향/양방향 통신 도메인.
- **채널 경계**:
  - **공지 채널 (Broadcast)**: 관리자가 전체 트랙(ACTIVE 상태)에 단방향 송신. 트랙별로 읽음 상태를 추적한다.
  - **트랙 다이렉트 (Track Direct)**: 관리자와 개별 트랙 간의 1:1 양방향 채널. 트랙 생성 시 자동으로 활성화되며, 진행자는 소속 트랙 스레드에만 접근할 수 있다.

---

## 3. 엔티티별 상태 머신

### 3.1 조(Group)의 코너별 방문 상태
```
[미방문 NOT_VISITED]
      │  (진행자가 시작 스캔 / 수동 처리 — 트랙이 비어있을 때만 성공)
      ▼
[진행중 IN_PROGRESS]
      │  (진행자가 종료 확인 처리)
      ▼
[완료 COMPLETED]  ── 재방문 시도 시 시스템이 거부(중복 방문 금지 규칙)
```

### 3.2 조(Group)의 전체 상태 (10개 코너 순회표 기반 파생 상태)
```
[대기중 IDLE_MOVING]  : 어느 코너에도 속하지 않고 이동 중
[진행중 AT_CORNER]    : 특정 코너에서 IN_PROGRESS 방문 존재
[완주 FINISHED]       : 10개 코너 모두 COMPLETED
```

### 3.3 트랙(Track) 생명주기 및 가동 상태
```
[미생성] ──(관리자가 트랙 생성)──▶ [ACTIVE·유휴 IDLE]
                                            │  (진행자가 시작 스캔/수동 처리)
                                            ▼
                                   [ACTIVE·동작중 BUSY] ──(종료 확인 처리)──▶ [ACTIVE·유휴 IDLE]
                                            │
                                            │(관리자가 삭제/교체 — IN_PROGRESS면 차단)
                                            ▼
                                         [DELETED] (PIN 무효화, 세션 종료)
```

### 3.4 PIN 로그인 실패 시 점증형 지연 (신뢰 기기 전용)
이미 신뢰 기기로 승인된 기기가 PIN 입력을 틀렸을 때 무차별 대입 방지를 위해 기기 단위 지연이 점증적으로 적용된다.
- 1~2회 실패: 즉시 재시도 가능
- 3회 실패: 5초 지연
- 4회 실패: 30초 지연
- 5회 실패 이상: 2분 지연 + 관리자 경고 알림
- 관리자는 대시보드에서 해당 기기의 실패 카운트와 지연 상태를 즉시 해제(Reset)할 수 있다.

## 4. 핵심 상호작용과 보안 경계

- **보안 경계**:
  - 관리자는 개인 계정으로, 진행자는 신뢰 기기에서 발급된 트랙 PIN으로만 세션을 획득한다.
  - 진행자 세션은 배정된 트랙 범위로 정보 접근 범위가 엄격히 제한된다.
  - **대리 스캔 위협 제외**: 다른 조의 배지 스캔 도용 위협은 이번 도메인 설계 범위에서 제외한다.
  - 모든 인증 성공/실패, 스캔, 규칙 변경, 기기 승인/철회, 트랙 관리는 감사 로그(Actor, Timestamp, Action, Target, Success/Fail)에 적재된다. (메시지 통신 내역은 감사 대상에서 제외)

## 5. 비즈니스 규칙 요약

1. **완전 순회 지향**: 모든 조가 모든 코너를 1회 완료하는 것을 지향하나, 프로그램 종료 시 미완료 조가 있어도 오류로 보지 않고 부분 완주로 종료한다.
2. **중복 방문 금지**: 조-코너 조합당 `COMPLETED` 방문은 캠프당 1회로 제한된다.
3. **수동 처리 허용**: QR 배지 손상 시 진행자가 직접 조를 선택해 입장을 수동 처리할 수 있으며, 이 때 방문의 입력 방식은 `MANUAL`로 기록되고 QR 스캔과 동일한 불변식이 적용된다.
4. **퇴장 확인 간소화**: 퇴장 시에는 재스캔을 요구하지 않으며, 트랙 내 단일 진행 중인 조에 대해 종료 확인 처리로 완료한다.
5. **단일 진행 제약**: 조는 동시에 둘 이상의 코너에서 `IN_PROGRESS` 상태일 수 없다.
6. **트랙 기반 동시성**: 코너 내 가동 트랙 수만큼 조가 동시 가동되며, 대기 관리(순서/인원)는 도메인 및 시스템 범위 밖에서 물리적으로 관리된다.
7. **트랙 불변 귀속**: 트랙이 살아있는 동안 소속 코너(`corner_id`)는 변경 불가능하다. 코너 변경은 트랙 교체(기존 트랙 만료 및 새 트랙 생성)로만 가능하다.
8. **트랙 삭제 제한**: 진행 중인 방문이 있는 트랙은 삭제나 교체가 하드 블록된다. 마지막 트랙 삭제로 코너가 미가동 상태가 되는 것은 경고 후 허용한다.
9. **기기 신뢰 기반**: 미등록 기기는 기기 등록 요청 및 관리자 승인을 거쳐 APPROVED 상태의 토큰을 획득해야만 트랙 PIN 입력 화면에 접근할 수 있다.
10. **세션 지속성**: 진행자 세션은 유휴 만료되지 않으며, 강제 로그아웃, 트랙 삭제, 캠프 종료, PIN 재발급 시에만 즉시(진행 중인 방문 무관) 만료된다.
11. **병목 판정의 비율화**: 병목은 절대 시간이 아닌 목표시간 대비 평균 편차 비율 및 최소 표본 파라미터를 기준으로 실시간 판정하며, 관리자가 실시간 변경 가능하다.
12. **캠프 생애주기 단방향성**: 캠프는 `PENDING` -> `ACTIVE` -> `ENDED`로 명시적 조작에 의해서만 단방향 전이되며, 종료된 캠프는 재사용할 수 없다.

## 6. 확인된 도메인 가정 (운영팀 확인 완료)

1. **다회차 운영 없음**: 캠프당 코너학습은 정확히 1회만 운영된다.
2. **필수 코너 없음(현재는)**: 모든 코너가 선택 순회 대상이다. (추후 확장을 위해 코너에 `is_mandatory` 속성만 유지)
3. **코너 간 순서 의존 없음**: 완전히 독립적인 자유 순환 구조이다.
4. **트랙 개설 가능 여부는 운영 단계에서 결정**: 사전에 트랙 수를 제한하지 않고 현장에서 실시간 생성/삭제한다.
5. **간식 시간 등 중단이 있어도 소요시간 계산에 영향 없음**: 방문 진행 중 중단에 대한 일시정지 개념을 두지 않는다.
6. **미완료(부분 완주) 조에 대한 후속 조치 없음**: 보충이나 이월 없이 그대로 기록을 보존하고 종료한다.
7. **진행자는 팀 단위, 트랙 겸임 없음**: 진행자 팀은 배정된 트랙에 고정되며 타 트랙을 겸임할 수 없다.
8. **대기열은 시스템 범위 밖**: "입장 스캔"과 "대기열 등록"을 완전히 분리하여 대기열은 물리적으로만 존재하고 시스템에서는 추적하지 않는다.
