5# Issue #142 진행자 공지 읽음 처리 Plan

## 0. 조사 결과 · 원인

Issue #142의 재현 경로는 **진행자 앱에서 공지함을 열어 읽음 POST가 204로 성공한 뒤에도, 메인 화면의 공지 아이콘 배지가 남는 것**이다.

- `frontend/lib/facilitator/features/broadcast_inbox/broadcast_inbox_screen.dart`는 공지함 진입 시 `readAt == null`인 각 공지에 `POST /messages/broadcast/{id}/read`를 호출하고, 완료 뒤 목록 provider를 invalidate한다.
- `backend/internal/usecase/announcement.go`의 `MarkNoticeRead`는 현재 세션의 트랙에 해당하는 `announcement_receipts.read_at`을 최초 한 번 저장한다. 따라서 쓰기 경로는 존재하고 멱등적이다.
- 하지만 `backend/internal/infrastructure/web/message_handler.go`의 `ListBroadcasts`는 `Announcement`만 조회해 `MessageResponse`를 조립한다. 이때 트랙별 영속 상태인 receipt의 `read_at`/`isRead`를 매핑하지 않아 응답의 `readAt`은 항상 `null`, `isRead`는 항상 `false`가 된다.
- 메인 헤더와 공지함은 모두 이 목록의 `readAt == null`을 미확인 기준으로 사용한다. invalidate 뒤에도 같은 스냅샷이 내려오므로 배지가 사라지지 않는다.

**해결 방향:** 기존 `MessageResponse.readAt`/`isRead` 필드의 의미를 TrackAuth 호출 시 “현재 진행자 트랙의 해당 공지 receipt 상태”로 확정하고, 목록 조회를 receipt와 한 번에 결합한다. 읽음 POST, 테이블 구조, 생성된 Flutter 모델은 이미 존재하므로 새 엔드포인트나 별도 미확인 카운터는 만들지 않는다.

## 2. 계약 및 객체 설계

### 2.1 API 계약

기존 경로와 응답 모델을 유지한다.

```yaml
# GET /camps/{campId}/messages/broadcast (TrackAuth)
# MessageResponse의 Broadcast 항목에서:
isRead: true              # 현재 인증된 트랙의 receipt.read_at 존재 여부
readAt: 2026-07-20T...Z   # 현재 인증된 트랙의 최초 읽음 시각, 미읽음이면 생략
```

- `GET /camps/{campId}/messages/broadcast`의 설명과 `MessageResponse` 필드 설명을 갱신해, 위 값이 **TrackAuth 응답에서 현재 트랙 기준**임을 명시한다.
- AdminAuth 호출에는 특정 트랙의 receipt를 임의로 선택해 넣지 않는다. 기존처럼 공지 자체 정보만 반환한다(`isRead=false`, `readAt` 생략). 관리자별 읽음 기능은 이번 범위가 아니다.
- 필드와 경로는 이미 OpenAPI에 있으므로 Flutter 생성 모델의 타입 변경은 없다. 다만 계약 변경은 `workflow/Collaborate.md`에 따라 프론트엔드가 이 계약을 명시한 PR을 먼저 열고, 백엔드가 Swagger 원본과 구현을 반영한다.

### 2.2 조회 모델 (CQRS)

`Announcement`는 캠프 공지 자체이고 `AnnouncementReceipt`는 트랙별 상태다. 화면이 요구하는 것은 두 값을 합친 읽기 전용 스냅샷이므로, 엔티티를 변형하거나 공지마다 receipt를 별도 조회하지 않는다. CQRS를 도입하는 이 변경에서는 명령 포트와 조회 포트를 완전히 분리한다.

```go
// /Users/lsjtop10/projects/cornermon/backend/internal/usecase/announcement.go
// 책임: TrackAuth 목록에 필요한 공지 본문과 현재 트랙의 receipt 상태를 함께 표현하는 읽기 DTO.
type BroadcastNoticeView struct {
    Announcement *domain.Announcement
    ReadAt       domain.Optional[time.Time]
}

// AnnouncementQueryService
func (s *AnnouncementQueryService) ListNoticesForTrack(
    ctx context.Context,
    campID domain.CampID,
    trackID domain.TrackID,
) ([]BroadcastNoticeView, error)
```

```go
// /Users/lsjtop10/projects/cornermon/backend/internal/usecase/port.go
// 명령 경로: 공지 생성만 담당한다.
type AnnouncementRepository interface {
    Save(ctx context.Context, announcement *domain.Announcement) error
}

// 조회 경로: 공지 본문과 수신 확인 상태를 읽기 전용 모델로 반환한다.
type AnnouncementQuerier interface {
    ListNoticesByCamp(ctx context.Context, campID domain.CampID) ([]*domain.Announcement, error)
    ListNoticeViewsByCampAndTrack(
        ctx context.Context,
        campID domain.CampID,
        trackID domain.TrackID,
    ) ([]BroadcastNoticeView, error)
    ListAnnouncementReceipts(
        ctx context.Context,
        announcementID domain.AnnouncementID,
    ) ([]*domain.AnnouncementReceipt, error)
}
```

`AnnouncementService`는 `SendAnnouncement`·`MarkNoticeRead`만 보유하는 명령 서비스로 남긴다. `AnnouncementQueryService`는 `AnnouncementQuerier`를 통해 `ListNoticesByCamp`·`ListNoticesForTrack`·`GetAnnouncementReceipts`를 제공하며, 명령 포트에 의존하거나 상태를 바꾸지 않는다. 웹 핸들러도 명령/조회 유스케이스 인터페이스를 별도로 주입받는다.

조회도 단순 데이터 반환이 아니다. `ListNoticesForTrack`는 middleware가 인증해 전달한 `trackID`가 요청 `campID`에 속하는지 `TrackRepository`와 `CornerRepository`로 확인한 뒤에만 `AnnouncementQuerier`를 호출한다. 일치하지 않으면 `domain.ErrTrackScopeForbidden`을 반환한다. 관리자 목록과 수신 확인 현황도 각 라우트의 AdminAuth 결과를 전제로 해당 조회 유스케이스만 호출한다. 따라서 인증 토큰 해석은 middleware가, 트랙-캠프 범위 인가는 조회 유스케이스가 맡는다.

Postgres는 `pgAnnouncementRepository`(명령)와 `pgAnnouncementQuerier`(조회)를 별도 어댑터로 제공한다. 조회 어댑터는 `announcements`와 `announcement_receipts`를 `announcement_id` 및 전달받은 `track_id`로 `LEFT JOIN`한 단일 sqlc 쿼리를 사용한다. receipt가 없거나 `read_at IS NULL`이면 `domain.None[time.Time]()`을 반환한다. 이는 공지 N건당 receipt N회 조회하는 N+1 방식보다 정확하고 일관된 목록 스냅샷이다.

## 4. 테스트 설계

### 백엔드

- `ShouldReturnTrackReceiptStateWhenFacilitatorListsBroadcasts`: 미읽음·읽음 공지가 섞인 경우, TrackAuth 목록의 각 `isRead`와 `readAt`이 해당 트랙 receipt와 정확히 일치한다.
- `ShouldNotExposeTrackReceiptWhenAdminListsBroadcasts`: AdminAuth 목록은 공지 자체만 반환하고 특정 트랙의 읽음 시각을 노출하지 않는다.
- `ShouldPreserveFirstReadAtWhenNoticeIsMarkedReadTwice`: 기존 `MarkNoticeRead`의 멱등성 회귀를 확인한다.
- `ShouldReturnUnreadWhenReceiptReadAtIsNull`: `LEFT JOIN` 결과의 nullable `read_at`이 미읽음으로 변환된다.
- `ShouldUseOnlyAuthenticatedTrackForBroadcastReceiptState`: 다른 트랙 receipt가 존재해도 현재 `facilitatorSession.TrackID()` 것만 반환한다.
- `ShouldRejectBroadcastQueryWhenAuthenticatedTrackIsOutsideCamp`: 인증된 트랙이 요청 캠프에 속하지 않으면 조회 유스케이스가 `ErrTrackScopeForbidden`을 반환하고 조회 포트를 호출하지 않는다.

### 프론트엔드

- 공지함 진입 시 미읽음 ID만 읽음 POST에 전달한다(기존 테스트 유지).
- 읽음 POST 완료 후 provider의 다음 응답이 `readAt`을 포함하면 `MainTrackHeader`의 공지 배지가 사라진다.
- 이미 `readAt`이 있는 공지는 재요청하지 않고 일반 글꼴로 표시한다.
- 읽음 POST 실패 시 실패를 관찰 가능하게 만들고, 성공한 것처럼 UI 상태를 확정하지 않는다.

## 5. 검증 체크리스트

### 아키텍처·보안

- [x] `domain` 패키지에 postgres/web 의존성을 추가하지 않는다.
- [x] `AnnouncementRepository`는 명령(`Save`)만 가지며, `AnnouncementQuerier` 이외의 포트에 조회 메서드를 추가하지 않는다.
- [x] `AnnouncementService`는 명령만, `AnnouncementQueryService`는 조회만 담당하며 서로의 포트에 의존하지 않는다.

- [x] 공지×receipt 조회에 별도 N+1 루프를 만들지 않는다.
- [x] 진행자 응답의 receipt는 middleware가 인증한 세션의 `TrackID`만 기준으로 한다.
