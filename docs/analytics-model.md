# 관리자 집계·통계 설계 (사후 분석 중심)

> §domain-model.md 기반. 여기서는 "무엇을 어떤 단위로 집계하는가"를 정의한다. 저장 방식(테이블/캐시 등)은 기술 설계에서 다룬다.

---

## 0. 설계 원칙

1. **모든 지표는 파생값이다.** Visit 레코드(§domain-model.md 2.6)와 감사 로그(트랙 생성/삭제/교체, 규칙 변경, 로그인 이벤트)만이 원장(source of truth)이며, 통계 수치를 별도로 수동 입력하거나 보정하지 않는다. 코너 상태를 트랙 상태의 집계로 파생시켰던 것(§3.3)과 동일한 원칙을 통계에도 그대로 적용한다.
2. **실시간 지표 vs 사후 지표는 다른 문제다.** 캠프 진행 중 대시보드는 "지금 이 순간의 스냅샷"(혼잡도, 유휴 여부 등 §3.3)만 있으면 되고 매 스캔마다 가볍게 재계산 가능하다. 반면 사후 분석은 "캠프 전체 기간에 걸친 집계"(평균, 표준편차, 누적 시간)가 필요해 계산 비용이 다르다. 이 문서는 후자를 중심으로 다룬다.
3. **캠프 스코프**: 모든 통계는 특정 캠프 ID(§2.0) 내에서 계산된다. 여러 캠프 비교는 캠프별 통계를 나란히 놓고 보는 것으로 처리하며, 캠프를 가로지르는 통합 집계 엔티티는 두지 않는다.

---

## 1. 집계 단위별 지표

### 1.1 캠프 레벨 (전체 요약)
| 지표 | 정의 |
|---|---|
| 전체 완주율 | 완주(FINISHED) 조 수 / 전체 조 수(20) |
| 전체 방문 완료율 | 완료된 방문 수 / 이론상 최대 방문 수(20조×10코너=200) |
| 부분 완주 조 목록 | 완주하지 못한 조와, 그 조가 못 간 코너 목록 |
| 프로그램 총 소요시간 | 첫 방문 started_at ~ 캠프 종료 선언 시각 |
| 전체 평균 목표시간편차 | 모든 COMPLETED 방문의 편차 평균 |
| 코너별 병목 랭킹 | 코너를 "평균편차" 기준 내림차순 정렬 (대기열 데이터가 없어 "혼잡 누적시간"은 산출 불가 — §domain-model.md 2.6-c). 이 랭킹은 컷오프 없이 전체 코너를 항상 보여준다 — 실시간 대시보드의 "병목" 이진 판정(최소 표본+비율 임계치, §domain-model.md 2.6-e)과는 별개다. |
| 수동 처리 비율 | input_method=MANUAL 방문 수 / 전체 방문 수 (QR 품질 이슈 파악용) |
| 규칙 변경 총 횟수 | 캠프 기간 중 Rule Override 이벤트 수, 코너별 분포 |
| 트랙 생성/삭제/교체 총 횟수 | 운영 중 트랙 구성이 얼마나 유동적이었는지 |
| 관리자 예외 승인(중복방문) 횟수 | §5-2 예외 처리가 얼마나 발생했는지 |

### 1.2 코너 레벨 (코너마다 1행)
| 지표 | 정의 |
|---|---|
| 처리 완료 조 수 | 이 코너에서 COMPLETED된 방문 수 (최대 20) |
| 미방문 조 수 및 목록 | 20 − 처리 완료 조 수, 어떤 조인지 |
| 평균/중앙값/표준편차 소요시간 | COMPLETED 방문들의 (ended_at − started_at) 분포 |
| 평균 목표시간편차, 편차>0 비율 | 목표시간을 초과한 방문의 비율 — "이 코너는 시간이 늘 부족했다"를 드러냄 |
| BUSY/유휴/미가동 누적시간 | 코너 상태(§3.3)가 각 상태였던 총 시간(감사 로그의 트랙 생성/삭제 타임스탬프 + Visit의 started/ended로 재구성). 대기열 길이 기반 "혼잡" 단계는 없음(§domain-model.md 2.6-c) |
| 트랙별 처리량 분포 | 같은 코너 내 트랙 간 처리량 편차 — 트랙별 진행자 역량/속도 차이를 드러낼 수 있음 |

> **주의**: 대기시간(줄서서 기다린 시간)은 이 시스템의 Visit 레코드에 존재하지 않는다(§domain-model.md 2.6-c) — 진행자는 트랙이 실제로 빌 때만 시작 스캔을 하기 때문에, "얼마나 오래 밀렸는지"는 코너의 BUSY 누적시간·평균편차로 간접 추정할 수밖에 없다.

### 1.3 트랙 레벨 (트랙마다 1행, DELETED된 과거 트랙 포함)
| 지표 | 정의 |
|---|---|
| 생명주기 구간 | 생성시각 ~ 삭제시각(또는 캠프 종료), 소속 코너 |
| 처리 완료 방문 수 | 이 트랙에서 COMPLETED된 방문 수 |
| 평균 소요시간 | 이 트랙만의 평균 (같은 코너 내 다른 트랙과 비교 가능) |
| PIN 로그인 실패 횟수 | 이 트랙에서 발생한 PIN 오입력 누적 (§3.4) — 특정 자리에서 유독 오입력이 잦았는지 |

> 트랙 교체(§2.7)로 코너가 바뀐 트랙은, 교체 전후가 서로 다른 트랙 레코드이므로 자동으로 구간별로 분리 집계된다 — 별도 처리 불필요.

### 1.4 조 레벨 (조마다 1행)
| 지표 | 정의 |
|---|---|
| 완주 여부 | FINISHED / PARTIAL |
| 완료 코너 수 | 0~10 |
| 코너별 소요시간 리스트 | 이 조가 각 코너에서 얼마나 걸렸는지 (개인화된 순회 기록) |
| 총 활동시간 | 완료한 모든 방문의 소요시간 합 |
| 미방문 코너 목록 | 부분 완주인 경우 |

### 1.5 시계열(타임라인) 지표 — 그래프용
| 지표 | 정의 |
|---|---|
| 시간대별 진행중 방문 수 | 예: 5분 버킷마다 IN_PROGRESS였던 방문 수 → 전체 처리량 추이 그래프 |
| 시간대별 누적 완료 방문 수 | 처리량이 시간에 따라 어떻게 늘어났는지(초반 vs 후반 속도 비교) |

이 지표들은 캠프 종료 후 "어느 시간대에 전체적으로 정체가 심했는가"를 보여줘서, 다음 캠프에서 "그 시간대 앞뒤로 코너 수/트랙 수를 조정하자" 같은 결정에 쓰인다.

### 1.6 운영/보안 지표 (부가, 감사 로그 기반)
| 지표 | 정의 |
|---|---|
| PIN 로그인 실패/성공 비율 | 신뢰 기기 기준 |
| 기기 등록 요청/승인/거절/회수 건수 | |
| 관리자별 조작 횟수 | 규칙 변경, 트랙 조작, 강제 로그아웃, 잠금 해제 등 — 감사 로그 §4.4 기반 |
| 트랙별 다이렉트 메시지 발신 횟수 | 어느 트랙이 관리자에게 도움 요청/메시지를 자주 보냈는지(§2.9) — 인력 배치나 코너 난이도 조정의 참고 자료 |
| 공지별 읽음 도달률 | 공지 발송 후 몇 개 트랙이 몇 분 만에 읽었는지 — 공지 전달 체계가 실제로 잘 작동했는지 사후 검증 |

## 2. 집계 시점과 갱신 방식

- **실시간(캠프 진행 중)**: 대시보드에 필요한 건 §3.3의 "현재 스냅샷"류(코너/트랙 상태: BUSY/IDLE/미가동)뿐이며, 매 스캔 이벤트마다 즉시 갱신되는 가벼운 계산이다. 이 문서의 §1.1~1.6 같은 전체 기간 집계를 실시간으로 매번 다시 계산할 필요는 없다.
- **사후(캠프 종료 시)**: 캠프 종료가 선언되는 시점에 §1의 모든 지표를 한 번 배치로 계산해 **"캠프 결과 리포트"**로 고정해둔다. 종료 후에는 Visit/감사 로그를 포함해 어떤 데이터도 더 이상 변경되지 않으므로(중복방문 예외 승인 등 사후 보정도 불가), 이 리포트는 재생성 없이 캠프의 최종 결과로 남는다.

## 3. 산출물

- **관리자 대시보드(실시간)**: §1.1~1.2의 스냅샷 버전만 — 지금 혼잡한 코너, 지금 유휴/미가동인 코너.
- **캠프 결과 리포트(사후, 다운로드 가능)**: §1의 전 지표를 담아 엑셀로 내보내기(§domain-model.md의 "PIN 목록 내보내기"와 같은 패턴) — 캠프가 끝난 뒤 다음 캠프를 기획할 담당자에게 전달할 산출물.
