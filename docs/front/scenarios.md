# 코너학습 시스템 — Given-When-Then 시나리오

> 전제(§domain-model.md 반영): 조 20, 코너 10, 트랙은 관리자가 생성/삭제하는 독립 엔티티(진행자 1:1, corner_id는 생성 후 불변), 관리자 2명(완전 동등 권한).
> 인증 모델: **진행자 = 트랙 생성 시 자동 발급되는 트랙 PIN**(트랙 삭제 시 즉시 무효화), **관리자 = 개인 계정(ID/비밀번호)**.
> "코너가 바뀐다"는 트랙의 corner_id를 수정하는 게 아니라 **기존 트랙 삭제 + 신규 트랙 생성("트랙 교체")**으로 처리한다.
> **대기열은 시스템 범위 밖(§domain-model.md 2.6-c)**: 진행자의 스캔은 "시작 스캔" 하나뿐이며, 트랙이 비어있을 때만 성공한다. 트랙이 BUSY면 시작 스캔 자체가 거부되고, 대기 관리는 현장에서 물리적으로 이뤄진다 — QUEUED 상태나 자동 승계는 없다.
> **종료는 QR 재스캔이 아니라 화면의 종료 확인이다(§domain-model.md 2.6-d)**: 한 트랙은 동시에 최대 1개의 `IN_PROGRESS` 방문만 가질 수 있으므로(§2.2), 종료 시점엔 "누구를 종료시키는지"가 이미 유일하게 정해져 있다 — 조 QR 배지를 다시 스캔해 신원을 재확인할 이유가 없다. 진행자는 화면의 종료 확인 버튼을 두 번 탭해 종료한다(첫 탭은 무장, 제한시간 안에 같은 버튼을 다시 탭해야 실제로 종료됨 — 시간이 아니라 의도적인 2회 확인으로 오조작을 막는다).
> 형식: Gherkin. 대리 스캔(배지 도용)은 §domain-model.md 4.4에 따라 이번 범위에서 의도적으로 제외.

## Feature 1: 조-코너 방문 처리 (핵심 도메인 흐름)

```gherkin
Feature: 조의 코너 방문 처리
  진행자가 조장의 QR 배지를 스캔해 방문을 시작하고,
  화면의 종료 확인(두 번 탭)으로 방문을 마친다 — 종료는 재스캔하지 않는다.
  시스템은 중복 방문을 막고 소요시간/편차를 계산한다.
  시작 스캔은 트랙이 비어있을 때만 성공한다(대기열 없음).

  Background:
    Given "3번 코너 1번 트랙" 진행자가 정상적으로 로그인되어 있다
    And "5조"는 아직 "3번 코너"를 방문한 기록이 없다

  Scenario: 정상적인 방문 시작과 종료
    Given "3번 코너 1번 트랙"이 유휴(IDLE) 상태이다
    When 진행자가 "5조"의 QR 배지를 시작 스캔한다
    Then "5조"의 "3번 코너" 방문 상태가 즉시 "IN_PROGRESS"로 바뀐다
    And 방문 시작시각이 현재 스캔 시각으로 기록된다
    And 방문 입력 방식(input_method)이 "QR_SCAN"으로 기록된다
    And "3번 코너 1번 트랙" 상태가 "BUSY"로 바뀐다
    When 10분 후 진행자가 화면의 종료 확인 버튼을 두 번 탭한다(재스캔 없음 — 트랙에 "5조" 하나만 진행 중이므로 재식별 불필요)
    Then "5조"의 "3번 코너" 방문 상태가 "COMPLETED"로 바뀐다
    And 소요시간이 10분으로 계산되어 저장된다
    And 목표시간편차(Deviation)가 0으로 계산된다
    And "5조"의 순회표에서 "3번 코너" 항목이 완료 처리된다
    And "3번 코너 1번 트랙" 상태가 다시 "IDLE"로 바뀌어 다음 조의 시작 스캔을 받을 수 있다

  Scenario: QR 스캔 불능 시 방문 시작과 종료
    Given "3번 코너 1번 트랙"이 유휴(IDLE) 상태이다
    And 5조의 QR이 물에 젖어 훼손됐다.
    When 진행자가 5조를 수동으로 시작 처리한다.
    Then "5조"의 "3번 코너" 방문 상태가 "IN_PROGRESS"로 바뀐다
    And 방문 시작시각이 현재 처리 시각으로 기록된다
    And 방문 입력 방식(input_method)이 "MANUAL"로 기록된다
    And "3번 코너 1번 트랙" 상태가 "BUSY"로 바뀐다
    When 10분 후 진행자가 화면의 종료 확인 버튼을 두 번 탭한다(배지가 계속 훼손 상태여도 무관 — 종료는 QR을 쓰지 않는다)
    Then "5조"의 "3번 코너" 방문 상태가 "COMPLETED"로 바뀐다
    And 소요시간이 10분으로 계산되어 저장된다
    And 목표시간편차(Deviation)가 0으로 계산된다
    And "5조"의 순회표에서 "3번 코너" 항목이 완료 처리된다

  Scenario: 종료 확인은 두 번 탭 — 첫 탭은 무장(arm)만, 시간 안에 재확인 필요
    Given "3번 코너 1번 트랙"이 "5조"를 처리 중(BUSY)이다
    When 진행자가 종료 확인 버튼을 한 번 탭한다
    Then 방문은 계속 "IN_PROGRESS" 상태를 유지한다
    And 버튼이 "다시 탭해 확인" 상태로 바뀐다
    When 정해진 시간(예: 2~3초) 안에 같은 버튼을 다시 탭한다
    Then 방문 상태가 "COMPLETED"로 바뀐다

  Scenario: 종료 확인 무장 상태를 시간 안에 재탭하지 않으면 자동 취소
    Given 진행자가 종료 확인 버튼을 한 번 탭해 "다시 탭해 확인" 상태가 되었다
    When 정해진 시간 안에 같은 버튼을 다시 탭하지 않는다
    Then 버튼은 원래 "종료 확인" 상태로 자동 복귀한다
    And 방문은 계속 "IN_PROGRESS" 상태로 남아있다 — 취소로 인한 별도 페널티나 기록은 없다

  Scenario: 이미 완료한 코너에 대한 중복 시작 스캔 거부
    Given "5조"가 "3번 코너" 방문을 이미 "COMPLETED"로 완료했다
    When 진행자가 "5조"의 QR 배지를 다시 시작 스캔한다
    Then 시스템은 스캔을 거부한다
    And 진행자 화면에 "이미 완료된 코너입니다" 경고가 표시된다
    And 새로운 방문 레코드는 생성되지 않는다

  Scenario: 관리자 승인을 통한 예외적 재방문 허용
    Given "5조"가 "3번 코너" 방문을 이미 "COMPLETED"로 완료했다
    And 관리자가 "5조"의 "3번 코너" 재방문을 예외 승인했다
    When 진행자가 "5조"의 QR 배지를 시작 스캔한다
    Then 시스템은 스캔을 허용한다
    And 새로운 방문 레코드가 생성되고 상태가 "IN_PROGRESS"가 된다
    And 이 예외 승인 행위가 감사 로그에 기록된다

  Scenario: 트랙이 이미 사용 중일 때 시작 스캔 거부 (대기열 없음)
    Given "3번 코너 1번 트랙"이 "8조"를 처리 중이라 "BUSY" 상태이다
    When 진행자가 "5조"의 QR 배지를 시작 스캔한다
    Then 시스템은 스캔을 거부한다 ("현재 진행중인 조가 있습니다" 메시지)
    And "5조"의 방문 레코드는 생성되지 않는다
    And "5조"가 몇 번째로 기다리고 있는지는 시스템에 기록되지 않는다 — 현장에서 물리적으로 순서를 관리한다
    When "8조"의 종료가 확인되어(종료 확인 두 번 탭) 트랙이 "IDLE"로 바뀐 뒤 진행자가 "5조"를 다시 시작 스캔한다
    Then 이번에는 스캔이 성공하고 "5조"의 방문이 "IN_PROGRESS"로 전환된다

  Scenario: 프로그램 종료 시각까지 일부 코너를 방문하지 못한 조의 처리
    Given 프로그램 종료 시각이 도래했다
    And "12조"는 10개 코너 중 8개만 "COMPLETED" 상태이다
    When 관리자가 프로그램 종료를 선언한다
    Then "12조"는 시스템 오류 없이 "부분 완주" 상태로 기록된다
    And 미완료 2개 코너 항목은 강제로 완료 처리되지 않고 그대로 미완료로 남는다
```

## Feature 2: 관리자 모니터링 및 규칙 변경

```gherkin
Feature: 관리자의 실시간 현황 파악 및 개입
  관리자는 전체 코너/트랙의 가동 상태(BUSY/IDLE/미가동)를 실시간으로 보고
  목표시간편차가 누적되는 병목 코너에 규칙 변경을 지시할 수 있다.

  Scenario: 전체 트랙 가동 상태 감지 (대기열 없이, 트랙 상태만으로)
    Given "3번 코너"의 모든 트랙이 "BUSY" 상태이다
    When 관리자가 대시보드를 조회한다
    Then "3번 코너"의 상태가 "전부 사용중"으로 표시된다 (대기 조 수는 시스템이 모르므로 표시되지 않는다)

  Scenario: 유휴 코너 감지
    Given "7번 코너"의 모든 트랙에 진행 중인 조가 없다
    When 관리자가 대시보드를 조회한다
    Then "7번 코너"의 상태가 "유휴"로 표시된다

  Scenario: 병목 코너에 대한 규칙 변경
    Given "3번 코너"가 지속적으로 목표시간편차 양수(+)를 기록하고 있다
    When 관리자가 "3번 코너"의 목표시간을 10분에서 7분으로 변경한다
    Then 이후 "3번 코너"에서 계산되는 목표시간편차는 7분 기준으로 산출된다
    And 이 규칙 변경 행위가 감사 로그에 "누가/언제/무엇을"과 함께 기록된다

  Scenario: 두 관리자의 동시 규칙 변경 충돌
    Given "3번 코너"의 목표시간이 10분이다
    When 관리자 A가 "3번 코너" 목표시간을 7분으로 변경한다
    And 관리자 B가 거의 동시에 "3번 코너" 목표시간을 8분으로 변경한다
    Then 나중에 반영된 변경(예: 관리자 B의 8분)이 최종값으로 적용된다
    And 두 변경 행위 모두 감사 로그에 각각 기록된다
```

## Feature 2-b: 행사 준비 — 트랙 생성/삭제와 PIN 내보내기

```gherkin
Feature: 트랙 생성/삭제 및 PIN 카드 목록 내보내기
  관리자는 행사 시작 전 코너마다 필요한 트랙을 생성하고(PIN 자동 발급),
  실물 PIN 카드 제작을 위해 현재 ACTIVE 트랙 목록을 파일로 내보낼 수 있다.

  Scenario: 코너에 트랙 생성
    Given "3번 코너"에 아직 트랙이 없다
    When 관리자가 "3번 코너"에 트랙을 1개 생성한다
    Then 새 트랙이 "ACTIVE" 상태로 생성되고 랜덤 6자리 숫자 PIN이 자동 발급된다
    And 이 생성 행위가 감사 로그에 기록된다

  Scenario: 같은 코너에 트랙 추가 생성 (동시 처리 용량 증설)
    Given "3번 코너"에 이미 ACTIVE 트랙이 1개 있다
    When 관리자가 "3번 코너"에 트랙을 1개 더 생성한다
    Then "3번 코너"는 이제 2개의 ACTIVE 트랙을 가지며 각각 별도의 PIN을 갖고 독립적으로 조를 처리한다

  Scenario: 현재 ACTIVE 트랙 PIN 목록 엑셀 내보내기
    Given 여러 코너에 걸쳐 ACTIVE 트랙이 총 14개 존재한다
    When 관리자가 "PIN 목록 내보내기"를 실행한다
    Then 코너명, 트랙 ID, PIN 값을 컬럼으로 가진 엑셀(xlsx) 파일이 다운로드된다
    And DELETED 상태인 과거 트랙은 목록에 포함되지 않는다
    And 이 내보내기 행위가 감사 로그에 기록된다

  Scenario: 진행 중인 방문이 있는 트랙 삭제 시도 차단
    Given "3번 코너 1번 트랙"이 "8조"를 처리 중(IN_PROGRESS)이다
    When 관리자가 이 트랙을 삭제하려 한다
    Then 삭제가 거부된다
    And "진행 중인 방문이 있어 삭제할 수 없습니다" 메시지가 표시된다

  Scenario: 코너의 마지막 트랙 삭제 시 확인 요구 (하드 블록 아님)
    Given "7번 코너"에 ACTIVE 트랙이 1개뿐이다
    When 관리자가 그 트랙을 삭제하려 한다
    Then 시스템은 "이 코너를 서비스할 트랙이 없어집니다" 경고를 표시하고 재확인을 요구한다
    And 관리자가 확인하지 않으면 삭제가 실행되지 않는다

  Scenario: 관리자 확인 후 코너가 미가동(0트랙) 상태로 전환 — 정상 허용
    Given "7번 코너"에 ACTIVE 트랙이 1개뿐이다
    When 관리자가 그 트랙을 삭제하고 경고에 대해 확인을 누른다
    Then 트랙이 "DELETED"로 삭제된다
    And "7번 코너"는 ACTIVE 트랙이 0개인 "미가동" 상태로 표시된다 (오류나 예외가 아니라 정상 상태)
    And 아직 "7번 코너"를 방문하지 못한 조들은 그 코너가 미가동임을 대시보드에서 확인할 수 있다

  Scenario: 정상적인 트랙 삭제 (유휴 상태)
    Given "7번 코너"에 트랙이 2개 있고, 그 중 1개는 유휴 상태(IN_PROGRESS 없음)이다
    When 관리자가 그 유휴 트랙을 삭제한다
    Then 트랙 상태가 "DELETED"로 바뀌고 PIN이 즉시 무효화된다
    And 이 삭제 행위가 감사 로그에 기록된다
```

## Feature 2-c: 코너 담당 변경 — 트랙 교체

> **진행자 앱 구현 현황(2026-07-11)**: 아래 "정상적인 트랙 교체" 시나리오의 자동 전환("계속하시겠습니까?" 확인 후 재PIN입력 없이 새 세션 전환)은 현재 API 계약에 이를 지원하는 신호가 없어 구현하지 않았다. 진행자 앱은 트랙 교체를 기존 트랙 삭제(`trackDeleted`)와 동일하게 처리해 세션을 강제 종료하고 재로그인을 요구한다. 자동 전환은 [이슈 #30](https://github.com/lsjtop10/cornermon/issues/30)에서 추적한다.

```gherkin
Feature: 트랙 교체 (기존 트랙 삭제 + 신규 코너로 트랙 생성)
  현장에서 "A코너를 담당하던 자리가 B코너를 담당하게" 바뀔 때,
  관리자는 트랙 교체 한 번으로 처리하고, 접속해 있던 진행자 기기는 재PIN입력 없이 자동 전환된다.

  Scenario: 정상적인 트랙 교체
    Given "3번 코너 1번 트랙"이 유휴 상태이고 그 트랙에 진행자 기기가 로그인되어 있다
    When 관리자가 이 트랙을 "5번 코너"로 교체 실행한다
    Then 기존 트랙("3번 코너")은 "DELETED"로 전환되고 PIN이 무효화된다
    And "5번 코너" 소속의 새 트랙이 "ACTIVE"로 생성되고 새 PIN이 자동 발급된다
    And 기존에 로그인돼 있던 진행자 기기에 "코너가 5번으로 변경되었습니다. 계속하시겠습니까?" 확인창이 표시된다
    When 진행자가 확인을 누른다
    Then 그 기기는 새 PIN을 입력하지 않고도 새 트랙("5번 코너") 세션으로 자동 전환된다
    And 이 교체 행위가 감사 로그에 기록된다

  Scenario: 진행 중인 방문이 있을 때 트랙 교체 차단
    Given "3번 코너 1번 트랙"이 "8조"를 처리 중(IN_PROGRESS)이다
    When 관리자가 이 트랙을 "5번 코너"로 교체하려 한다
    Then 교체가 거부된다
    And "진행 중인 방문이 완료된 후 다시 시도하세요" 메시지가 표시된다

  Scenario: 교체로 인해 원래 코너의 서비스가 완전히 끊기는 경우 경고 (확인 시 진행 허용)
    Given "3번 코너"에 ACTIVE 트랙이 1개뿐이다
    When 관리자가 그 트랙을 "5번 코너"로 교체하려 한다
    Then 시스템은 "3번 코너를 서비스할 트랙이 없어집니다" 경고를 표시하고 재확인을 요구한다
    And 관리자가 확인하지 않으면 교체가 실행되지 않는다
    When 관리자가 확인을 누른다
    Then 교체가 진행되고 "3번 코너"는 ACTIVE 트랙 0개인 "미가동" 상태가 된다 (정상 허용된 상태)

  Scenario: 교체 후 새로운 기기로 로그인 시 새 PIN 필요
    Given "5번 코너"로 막 교체된 새 트랙이 있고 새 PIN이 발급되어 있다
    And 아직 새 PIN 카드가 현장에 인쇄되어 비치되지 않았다
    When 다른 진행자 기기가 이 트랙에 로그인을 시도한다
    Then 새로 발급된 PIN을 입력해야만 로그인에 성공한다 (기존 트랙의 옛 PIN으로는 로그인 불가)
```

## Feature 2-d: 초기 설정 마법사 — 캠프/코너/트랙 일괄 생성 (A0-b)

```gherkin
Feature: 초기 설정 마법사
  관리자는 캠프 목록(A0-c)에서 "+ 새 캠프 시작"을 눌러 이 마법사로 진입한다(변경 —
  기존에는 캠프가 하나도 없는 최초 실행 시 로그인 직후 자동으로 이 화면에 떨어졌으나,
  이제 로그인 후에는 캠프 개수와 무관하게 항상 캠프 목록을 먼저 거친다). 여기서
  관리자는 코너를 하나씩 만드는 대신 이름을 줄바꿈으로 붙여넣어 한 번에 여러 코너와
  그 초기 트랙을 생성한다. 코너별 세부 조정(이름 정정, 목표시간·트랙 수 개별화)은

  Scenario: 캠프 정보 입력 후 다음 단계로 진행
    Given 관리자가 캠프 목록에서 "+ 새 캠프 시작"을 눌러 마법사에 진입했다
    When 관리자가 캠프 이름 "2026 여름 수련회"와 시작일/종료일을 입력하고 "다음"을 누른다
    Then 캠프 정보가 임시 저장되고 코너·트랙 일괄 생성 단계로 이동한다

  Scenario: 코너 이름 붙여넣기로 일괄 생성
    Given 관리자가 코너·트랙 일괄 생성 단계에 있다
    When 관리자가 코너 이름 10개를 줄바꿈으로 붙여넣고 기본 목표시간 10분, 코너당 기본 트랙 수 1개를 지정한다
    And 검토 단계에서 "설정 완료 → 대시보드로"를 누른다
    Then 코너 10개가 각각 목표시간 10분으로 생성된다
    And 코너마다 트랙이 1개씩 생성되고 각각 6자리 숫자 PIN이 자동 발급된다
    And 관리자는 대시보드로 이동하며 새로 생성된 코너 10개가 카드로 표시된다

  Scenario: 예시 템플릿으로 빠르게 채우기
    Given 관리자가 코너·트랙 일괄 생성 단계에 있고 아직 아무 이름도 입력하지 않았다
    When 관리자가 "예시 10개로 빠르게 시작" 버튼을 누른다
    Then 텍스트 영역에 표준 코너 이름 10개가 자동으로 채워진다
    And 관리자는 이름을 그대로 쓰거나 필요한 만큼 수정한 뒤 진행할 수 있다

  Scenario: 코너를 하나도 추가하지 않으면 다음 단계로 진행할 수 없음
    Given 관리자가 코너·트랙 일괄 생성 단계에서 코너 이름을 하나도 입력하지 않았다
    When 관리자가 "다음"을 누른다
    Then 진행이 차단되고 "코너를 1개 이상 추가하세요" 메시지가 표시된다
    And 검토 단계로 넘어가지 않는다

  Scenario: 로그인은 캠프 개수와 무관하게 항상 캠프 목록으로 이동한다(변경)
    Given 캠프와 코너가 이미 설정되어 있다
    When 관리자가 로그인한다
    Then 초기 설정 마법사가 아니라 캠프 목록 화면으로 이동한다
    And 마법사는 캠프 목록에서 "+ 새 캠프 시작"을 눌러야만 진입할 수 있다
```

## Feature 2-e: 트랙 일괄 관리 (A2B)

```gherkin
Feature: 여러 코너에 걸친 트랙 일괄 수정/삭제
  관리자는 서로 다른 코너에 속한 트랙을 한 화면에서 체크박스로 골라
  목표시간을 한 번에 바꾸거나 한꺼번에 삭제할 수 있다.
  삭제는 단건 삭제 가드①(§domain-model.md 2.2)을 선택 집합 단위로 그대로 적용한다 —
  선택 중 하나라도 진행 중인 방문이 있으면 전체를 거부하고 부분 삭제는 하지 않는다.

  Scenario: 여러 코너의 트랙을 선택해 목표시간 일괄 변경
    Given 관리자가 트랙 일괄 관리 화면에서 서로 다른 코너에 속한 트랙 3개를 선택했다
    When 관리자가 목표시간을 "8"분으로 입력하고 "적용"을 누른다
    Then 선택된 트랙이 속한 코너들의 목표시간이 모두 8분으로 변경된다
    And 이후 그 코너들에서 계산되는 목표시간편차는 8분 기준으로 산출된다

  Scenario: 선택한 트랙이 모두 유휴 상태면 일괄 삭제된다
    Given 관리자가 트랙 3개를 선택했고 셋 다 IDLE 상태다
    When 관리자가 "선택 삭제"를 누른다
    Then 선택된 3개 트랙이 모두 DELETED로 삭제되고 PIN이 즉시 무효화된다

  Scenario: 선택 항목 중 하나라도 BUSY면 전체 삭제가 차단된다
    Given 관리자가 트랙 3개를 선택했고 그 중 1개는 IN_PROGRESS 방문이 있어 BUSY 상태다
    Then "선택 삭제" 버튼이 비활성화되어 있다
    And "진행 중인 방문이 있어 삭제할 수 없습니다" 경고가 표시된다
    And 나머지 2개(IDLE)를 포함해 3개 트랙 중 어느 것도 삭제되지 않는다 — 부분 삭제는 하지 않는다

  Scenario: 상태 필터로 BUSY 트랙만 걸러 보기
    Given 트랙 일괄 관리 화면에 IDLE 트랙과 BUSY 트랙이 섞여 있다
    When 관리자가 상태 필터를 "BUSY만"으로 선택한다
    Then BUSY 상태인 트랙만 목록에 표시되고 결과 건수가 갱신된다

  Scenario: "전체 선택"은 필터로 화면에 보이는 트랙만 대상으로 한다
    Given 관리자가 상태 필터를 "IDLE만"으로 선택해 일부 트랙만 표시되고 있다
    When 관리자가 "전체 선택" 체크박스를 누른다
    Then 화면에 보이는 IDLE 트랙만 선택되고, 필터로 숨겨진 BUSY 트랙은 선택되지 않는다
```

## Feature 2-g: 캠프 목록 — 진행상황 분석과 사후 분석 플로우 분리 (A0-c)

```gherkin
Feature: 캠프 목록
  로그인 직후 여러 캠프(진행 중/준비 중/종료됨)를 한눈에 보고 고를 수 있는 지점을 둔다.
  진행 중 캠프는 운영 전체 화면으로, 종료된 캠프는 리포트 전용 화면으로 들어가며
  둘을 서로 다른 사이드바 세트로 명확히 분리한다(§domain-model.md 2.0-a).

  Background:
    Given 관리자로 로그인했다

  Scenario: 캠프가 하나도 없어도 로그인 후 캠프 목록으로 이동한다(변경)
    Given 캠프가 하나도 없다
    When 관리자가 로그인에 성공한다
    Then 초기 설정 마법사(A0-b)가 아니라 캠프 목록 화면으로 이동한다
    And 캠프 목록은 "아직 캠프가 없습니다" empty state와 "+ 새 캠프 시작" 버튼을 보여준다

  Scenario: 캠프가 하나라도 있으면 로그인 후 캠프 목록으로 이동한다
    Given 캠프가 하나 이상 존재한다
    When 관리자가 로그인에 성공한다
    Then 대시보드가 아니라 캠프 목록 화면으로 이동한다
    And 진행 중 캠프는 "진행 중" 배지, 종료된 캠프는 "종료됨" 배지로 표시된다

  Scenario: 진행 중 캠프를 열면 운영 사이드바 전체가 나타난다
    Given 캠프 목록에 "진행 중" 배지가 붙은 캠프가 있다
    When 관리자가 그 캠프를 클릭한다
    Then 대시보드로 이동하며 사이드바에 대시보드·조 현황·기기 관리·메시지·리포트·감사 로그·설정 7개 항목이 모두 보인다
    And 상단 바에 "코너학습 종료" 버튼이 보인다

  Scenario: 종료된 캠프를 열면 리포트 전용 축소 사이드바가 나타난다
    Given 캠프 목록에 "종료됨" 배지가 붙은 캠프가 있다
    When 관리자가 그 캠프를 클릭한다
    Then 리포트(A12) 화면으로 이동하며 실제 집계 데이터가 곧장 표시된다(재생성 없음)
    And 사이드바에는 "리포트" 항목만 보이고 조 현황·기기 관리·메시지·감사 로그·설정은 보이지 않는다
    And 상단 바에 "코너학습 종료" 버튼이 보이지 않는다

  Scenario: 코너학습 종료 후 대시보드가 아니라 캠프 목록으로 돌아간다
    Given 관리자가 진행 중 캠프의 대시보드에서 "코너학습 종료"를 확정했다
    When 종료 처리가 완료된다
    Then 그 캠프의 대시보드에 머무르지 않고 캠프 목록 화면으로 돌아간다
    And 방금 끝난 캠프가 목록에 "종료됨" 배지로 나타난다

  Scenario: "+ 새 캠프 시작"으로 새 캠프를 만들면 곧장 그 캠프로 진입한다
    Given 관리자가 캠프 목록 화면에 있다
    When 관리자가 "+ 새 캠프 시작"을 눌러 초기 설정 마법사를 완료한다
    Then 새 캠프가 "진행 중" 상태로 캠프 목록에 추가된다
    And 마법사 완료 즉시 그 캠프의 대시보드로 이동한다(캠프 목록을 다시 거치지 않음)

  Scenario: 초기 설정 마법사 1단계에서 캠프 목록으로 취소할 수 있다
    Given 캠프가 이미 하나 이상 있는 상태에서 관리자가 "+ 새 캠프 시작"으로 마법사에 들어왔다
    And 마법사 1단계(캠프 정보)에 있다
    When 관리자가 "← 캠프 목록" 버튼을 누른다
    Then 새 캠프를 만들지 않고 캠프 목록 화면으로 돌아간다
```

```gherkin
Feature: QR 배지 사전 생성과 조 등록
  현장에서 급하게 QR을 인쇄해 스코어판에 붙일 시간이 없으므로, 배지를 조보다 먼저
  대량으로 만들어 미리 부착해두고, 실제 조가 정해지면 그때 배지를 조에 등록한다
  (§domain-model.md 2.1-a). 초기 설정 마법사(A0-b)는 캠프·코너·트랙만 다루며
  조와 배지는 이 Feature로 별도로 다룬다.

  Background:
    Given 관리자로 로그인했다

  Scenario: 캠프 선택 없이 QR 배지를 대량 생성한다
    Given 관리자가 캠프 목록에서 "QR 배지 관리"로 들어왔다
    When 관리자가 생성 수량에 "40"을 입력하고 "배지 생성"을 누른다
    Then 미배정(UNASSIGNED) 배지 40장이 새로 생성된다
    And 이 동작은 어떤 캠프를 선택했는지와 무관하게 수행된다

  Scenario: 미배정 배지를 스티커 PDF로 내보낸다
    Given 미배정 배지가 여러 장 있다
    When 관리자가 "스티커 PDF로 내보내기"를 누른다
    Then 미배정 배지 전체가 스티커 인쇄용 PDF로 다운로드된다
    And 실제 인쇄는 이 PDF를 받아 다른 컴퓨터에서 진행한다(iPad에서 직접 인쇄하지 않음)

  Scenario: 카메라로 스캔해 조를 등록한다
    Given 관리자가 진행 중 캠프의 조 현황(A5) 화면에서 "+ 조 등록"을 눌렀다
    And "카메라로 스캔" 탭에 있다
    When 스코어판에 부착된 미배정 배지를 스캔하고 조 이름에 "21조"를 입력한 뒤 "등록 확정"을 누른다
    Then "21조"가 새로 생성되어 조 현황 목록에 나타난다
    And 방금 스캔한 배지는 배정됨(ASSIGNED) 상태로 바뀐다

  Scenario: 목록에서 골라 조를 등록한다
    Given 관리자가 "+ 조 등록" 모달의 "목록에서 선택" 탭에 있다
    When 미배정 배지 목록에서 배지 ID를 고르고 조 이름을 입력해 "등록 확정"을 누른다
    Then 그 배지로 새 조가 생성되며, 카메라 스캔으로 등록했을 때와 동일한 결과가 된다

  Scenario: 조 이름 없이는 등록을 확정할 수 없다
    Given 관리자가 배지를 스캔했거나 목록에서 선택했다
    But 조 이름 입력란이 비어 있다
    Then "등록 확정" 버튼은 비활성 상태로 유지된다

  Scenario: 이미 이번 캠프에 등록된 배지는 다시 등록할 수 없다
    Given 어떤 배지가 이번 캠프에서 이미 "5조"로 등록(ASSIGNED)되어 있다
    When 관리자가 그 배지를 다시 스캔하거나 목록에서 선택해 등록을 시도한다
    Then 등록이 거부되고 "이미 5조로 등록된 배지입니다" 안내가 표시된다

  Scenario: 미배정 배지로는 실제 방문 시작 스캔을 할 수 없다
    Given 어떤 배지가 아직 등록되지 않아 미배정(UNASSIGNED) 상태다
    When 진행자가 그 배지를 트랙에서 방문 시작 스캔으로 사용하려 한다
    Then 스캔이 거부되고 "이 QR은 아직 조로 등록되지 않았습니다" 안내가 표시된다
```

## Feature 2-i: 준비 중 캠프와 명시적 시작 (A0-c, A0-e)

```gherkin
Feature: 캠프 상태를 관리자의 명시적 액션으로만 전환
  캠프가 "코너·트랙이 준비됐으니 자동으로 시작"되지 않도록, PENDING→ACTIVE
  전이는 오직 관리자가 "코너학습 시작"을 확정하는 행위 하나로만 일어난다
  (§domain-model.md 2.0-a, 2.4-e). 그 전까지는 PIN이 발급돼 있어도 로그인이
  거부된다.

  Background:
    Given 관리자로 로그인했다

  Scenario: 초기 설정 마법사를 끝내면 ACTIVE가 아니라 PENDING으로 추가된다
    Given 관리자가 초기 설정 마법사에서 캠프·코너·트랙 설정을 완료했다
    Then 새 캠프가 "준비 중(PENDING)" 상태로 캠프 목록에 추가된다
    And 곧장 코너·트랙 준비 화면(A2B)으로 이동한다
    And 사이드바에는 코너·트랙/조 현황/기기 관리/설정 4개 항목만 보인다

  Scenario: 준비 중 캠프는 코너·트랙이 다 갖춰져도 저절로 시작되지 않는다
    Given 코너와 트랙이 모두 준비된 "준비 중" 캠프가 있다
    When 관리자가 아무 조작도 하지 않고 캠프 목록으로 돌아간다
    Then 그 캠프는 여전히 "준비 중" 배지로 남아있다

  Scenario: 준비 중 캠프에서 PIN으로 로그인할 수 없다
    Given "준비 중" 캠프의 어떤 트랙에 PIN이 이미 발급되어 있다
    When 진행자가 그 PIN으로 로그인을 시도한다
    Then 로그인이 거부되고 "코너학습이 아직 시작되지 않았습니다" 안내가 표시된다

  Scenario: "코너학습 시작"을 확정하면 그 즉시 PENDING에서 ACTIVE로 전환된다
    Given 관리자가 "준비 중" 캠프의 준비 모드 화면에 있다
    When 관리자가 "코너학습 시작"을 누르고 확인 모달에서 "시작 확정"을 누른다
    Then 캠프 상태가 "진행 중(ACTIVE)"으로 바뀐다
    And 사이드바가 운영 모드(대시보드·조 현황·기기 관리·메시지·리포트·감사 로그·설정 7개)로 전환된다
    And 대시보드(A1)로 이동한다

  Scenario: 시작 확정 이후에는 그 트랙의 PIN 로그인이 허용된다
    Given 관리자가 어떤 캠프의 "코너학습 시작"을 확정했다
    When 진행자가 그 캠프의 한 트랙 PIN으로 로그인을 시도한다
    Then 로그인이 정상적으로 허용된다
```

## Feature 2-j: PIN 재발급 (트랙 교체와 구분되는 별도 오퍼레이션)

```gherkin
Feature: PIN 재발급
  PIN 노출/오염이 의심될 때, 관리자는 트랙을 삭제/재생성하지 않고
  그 트랙의 PIN 값만 새로 뽑는다(§domain-model.md 2.5-c). 트랙 ID·소속 코너·
  방문 기록·다이렉트 스레드는 그대로 유지된다.

  Scenario: PIN 재발급은 PIN 값만 바꾸고 트랙 정체성은 그대로 유지한다
    Given "3번 코너 1번 트랙"의 PIN이 "482169"이다
    When 관리자가 이 트랙의 PIN을 재발급한다
    Then 트랙 ID, 소속 코너, 트랙 번호는 변경되지 않는다
    And 그 트랙에 쌓인 방문 기록과 다이렉트 스레드도 그대로 유지된다
    And 새로운 6자리 PIN이 발급되고 기존 PIN "482169"는 즉시 무효화된다

  Scenario: PIN 재발급은 로그인돼 있던 세션을 원자적으로 강제 종료한다
    Given "3번 코너 1번 트랙"에 진행자 세션이 로그인되어 있다
    When 관리자가 이 트랙의 PIN을 재발급한다
    Then 그 세션은 별도 API 호출 없이 즉시 종료된다
    And 진행자 화면은 PIN 로그인 화면(B1)으로 강제 전환된다

  Scenario: BUSY 상태여도 유예 없이 즉시 세션이 끊긴다
    Given "3번 코너 1번 트랙"이 어떤 조를 처리 중(BUSY)이다
    When 관리자가 이 트랙의 PIN을 재발급한다
    Then 처리 중이던 방문이 끝나기를 기다리지 않고 즉시 세션이 종료된다
    And 중단된 방문은 미완료 상태로 남는다

  Scenario: 재발급 이후에는 새 PIN으로만 로그인할 수 있다
    Given "3번 코너 1번 트랙"의 PIN이 "482169"에서 "917305"로 재발급되었다
    When 진행자가 옛 PIN "482169"로 로그인을 시도한다
    Then 로그인이 거부된다
    When 진행자가 새 PIN "917305"로 로그인을 시도한다
    Then 로그인이 성공한다

  Scenario: PIN 재발급은 트랙 교체와 다른 액션으로 감사 로그에 남는다
    Given 관리자가 어떤 트랙의 PIN을 재발급했다
    When 관리자가 감사 로그를 조회한다
    Then 이 행위는 "PIN 재발급"으로 기록되며 "트랙 삭제"나 "트랙 교체"와 구분된다
```

## Feature 3: 진행자 인증 (트랙 PIN)

```gherkin
Feature: 진행자 인증
  진행자는 개인 계정이 아니라 트랙 생성 시 자동 발급된 PIN만으로 인증하며,
  PIN 값 자체가 어느 트랙(및 코너)인지를 특정하므로 별도로 코너/트랙을 선택할 필요가 없다.
  단, PIN 화면 자체는 서버가 발급한 신뢰 토큰을 가진 "신뢰 기기"만 볼 수 있다(Feature 3-b) —
  미등록 기기는 PIN을 시도할 수조차 없는 하드 블록 대상이다.

  Scenario: 신뢰 기기의 정상 로그인
    Given 이 기기는 서버로부터 유효한 신뢰 토큰을 발급받은 "신뢰 기기"이다
    And "3번 코너 1번 트랙"이 ACTIVE 상태이고 PIN "482169"가 발급되어 있다
    When 진행자가 PIN 입력 화면에서 PIN "482169"를 입력한다
    Then 시스템이 PIN으로 트랙을 특정해 로그인이 성공하고 "3번 코너 1번 트랙" 스코프의 세션이 발급된다
    And 메인 화면(B2)으로 곧장 가지 않고 "3번 코너 · 1번 트랙이 맞습니까?" 확인 모달(B1-b)이 먼저 뜬다

  Scenario: PIN 로그인 직후 코너·트랙 확인
    Given PIN 로그인이 성공해 "3번 코너 · 1번 트랙이 맞습니까?" 확인 모달이 떠 있다
    When 진행자가 "예, 맞습니다"를 누른다
    Then 메인 화면(B2)으로 진입한다
    And 진행자 화면에는 "3번 코너 1번 트랙"의 현재 상태(IDLE/BUSY)만 노출된다

  Scenario: 확인 모달에서 잘못된 배정을 거부하면 재로그인으로 돌아간다
    Given PIN 로그인이 성공해 확인 모달이 떠 있지만 실제로는 다른 코너를 맡기로 한 팀이 잘못된 PIN 카드로 로그인했다
    When 진행자가 "아니요, 다시 로그인"을 누른다
    Then 방금 발급된 세션이 즉시 폐기된다
    And PIN 로그인 화면(B1)으로 돌아가며 메인 화면(B2)에는 진입하지 않는다

  Scenario: 미등록 기기는 PIN 화면 자체에 도달할 수 없음 (하드 블록)
    Given 이 기기는 유효한 신뢰 토큰이 없는(미등록) 기기이다
    When 앱을 실행한다
    Then PIN 입력 화면이 아니라 "기기 등록 대기 중" 화면이 표시된다
    And 이 상태에서는 어떤 PIN도 입력하거나 시도할 수 없다

  Scenario: 잘못된 PIN 입력 (신뢰 기기)
    Given 이 기기는 신뢰 토큰을 가진 신뢰 기기이다
    And 어떤 ACTIVE 트랙의 PIN과도 일치하지 않는 값이다
    When 진행자가 PIN "000000"을 입력한다
    Then 로그인이 거부된다
    And 세션이 발급되지 않는다
    And 이 실패 시도가 감사 로그에 기록된다

  Scenario: 삭제된 트랙의 옛 PIN으로 로그인 시도
    Given 이 기기는 신뢰 기기이다
    And "3번 코너 1번 트랙"이 삭제(DELETED)되어 PIN "482169"가 무효화되었다
    When 진행자가 PIN "482169"를 입력한다
    Then 로그인이 거부된다 (더 이상 어떤 ACTIVE 트랙과도 일치하지 않음)

  Scenario: 신뢰 기기의 연속 실패 - 점증형 지연
    Given 이 기기는 신뢰 토큰을 가진 신뢰 기기이다
    And 이 기기에서 PIN 입력이 2회 연속 실패했다
    When 3번째 로그인 시도가 실패한다
    Then 5초 지연이 적용된다 (즉시 차단되지 않음)
    When 이후에도 계속 실패해 4회, 5회에 도달한다
    Then 지연은 30초, 이후 2분으로 점점 늘어난다
    And 5회 이상부터는 관리자에게 알림이 표시된다

  Scenario: 세션 스코프 위반 - 타 코너 데이터 접근 차단
    Given "3번 코너 1번 트랙" 진행자로 로그인된 세션이 있다
    When 이 세션으로 "5번 코너"의 방문 데이터 조회를 시도한다
    Then 요청이 거부된다 (403 권한 없음에 준하는 처리)
    And 이 비정상 접근 시도가 감사 로그에 기록된다

  Scenario: 세션은 유휴 시간과 무관하게 만료되지 않는다
    Given 진행자 세션이 로그인된 지 오랜 시간(예: 6시간)이 지났고 그 사이 스캔이 없었다
    When 진행자가 스캔을 시도한다
    Then 시스템은 세션이 여전히 유효하다고 판단하고 스캔을 정상 처리한다
    And 재로그인(PIN 재입력)을 요구하지 않는다

  Scenario: 관리자의 명시적 강제 로그아웃만 세션을 끊는다
    Given "3번 코너 1번 트랙" 진행자로 로그인된 세션이 있다
    When 관리자가 이 세션을 강제 로그아웃시킨다
    Then 그 세션은 즉시 종료되고 이후 스캔 시도는 거부된다
    And 진행자는 트랙 PIN을 다시 입력해야 한다
    And 이 강제 로그아웃 행위가 감사 로그에 기록된다

  Scenario: 트랙 삭제로 인한 세션 강제 종료
    Given "3번 코너 1번 트랙" 진행자로 로그인된 세션이 있다
    When 관리자가 그 트랙을 삭제한다
    Then 그 세션은 즉시 종료되어 메인 화면(B2)에서 PIN 로그인 화면(B1)으로 강제 전환된다
    And "이 트랙이 삭제되어 로그아웃되었습니다" 안내가 표시된다

  Scenario: 코너학습 종료로 인한 세션 강제 종료
    Given "3번 코너 1번 트랙" 진행자로 로그인된 세션이 있다
    When 관리자가 "코너학습 종료"를 확정한다
    Then 그 세션은 즉시 종료되어 PIN 로그인 화면(B1)으로 강제 전환된다
    And "코너학습이 종료되어 로그아웃되었습니다" 안내가 표시된다

  Scenario: 세션 강제 종료는 BUSY 중에도 유예 없이 즉시 적용된다
    Given "3번 코너 1번 트랙"이 어떤 조를 처리 중(BUSY)이다
    When 트랙 삭제·강제 로그아웃·코너학습 종료 중 하나가 발생한다
    Then 그 조의 방문 처리가 끝나기를 기다리지 않고 즉시 PIN 로그인 화면(B1)으로 전환된다
    And 중단된 방문은 미완료 상태로 남고, 이를 이어서 완료 처리하는 별도 절차는 없다

  Scenario: 관리자의 즉시 잠금 해제
    Given 어떤 기기가 PIN 오입력 누적으로 지연(잠금)이 걸려 있다
    When 관리자가 대시보드에서 그 기기의 "잠금 해제"를 실행한다
    Then 실패 카운트가 즉시 초기화되고 지연이 풀린다
    And 그 기기는 곧바로 다시 PIN을 시도할 수 있다
    And 이 잠금 해제 행위가 감사 로그에 기록된다
```

## Feature 3-b: 기기 등록 (사전 신뢰 기기 승인)

```gherkin
Feature: 진행자 기기 등록 (요청 즉시 토큰 발급, 승인은 상태 전환)
  등록을 요청하는 순간 서버가 토큰을 즉시 발급해 클라이언트가 보안 저장소에 저장해두지만,
  그 토큰은 관리자가 승인해 "APPROVED" 상태로 바뀌기 전까지는 PIN 화면 접근 권한이 없다.
  신뢰 판단은 클라이언트가 자기 신고하는 기기정보가 아니라,
  오직 "서버 DB에서 APPROVED 상태인 토큰을 유효하게 제시하는가"로만 이뤄진다.

  Scenario: 등록 요청과 동시에 토큰 발급 (PENDING 상태)
    Given 관리자가 발급한 "기기 등록 코드"가 진행자 팀에게 공유되어 있다
    When 진행자 팀의 앱이 이 코드로 등록을 요청한다
    Then 서버는 그 즉시 무작위 신뢰 토큰을 생성해 응답으로 내려준다
    And 그 토큰은 서버 DB에 "PENDING" 상태로 기록된다
    And 클라이언트는 이 토큰을 기기 보안 저장소(Keychain/Keystore)에 저장한다 (평문 저장 금지)
    And 관리자 화면의 대기 목록에 이 요청이 나타난다
    And 이 시점에는 아직 PIN 화면에 접근할 수 없다 ("기기 등록 대기 중" 화면만 표시)

  Scenario: 관리자의 등록 승인 - 이미 발급된 토큰의 상태만 전환
    Given "PENDING" 토큰을 가진 기기 등록 요청이 있다
    When 관리자가 이 요청을 승인한다
    Then 서버는 그 토큰의 상태를 "PENDING"에서 "APPROVED"로 바꾼다 (새 토큰을 추가로 발급하지 않음)
    And 그 기기는 이미 저장해둔 동일한 토큰으로 PIN 화면에 접근할 수 있게 된다
    And 이 승인 행위가 감사 로그에 기록된다

  Scenario: 관리자의 등록 거절
    Given "PENDING" 토큰을 가진 기기 등록 요청이 있다
    When 관리자가 이 요청을 거절한다
    Then 그 토큰의 상태가 "REJECTED"로 고정된다
    And 그 기기는 계속 "기기 등록 대기 중" 화면에 머무른다 (PIN 화면 접근 불가)
    And 다시 시도하려면 새 등록 요청(새 토큰 발급)이 필요하다
    And 이 거절 행위가 감사 로그에 기록된다

  Scenario: 행사 중 예비 기기의 긴급 등록
    Given 기존 기기가 고장나 예비 기기를 급히 투입해야 한다
    When 예비 기기가 등록 코드로 등록을 요청해 즉시 PENDING 토큰을 발급받는다
    And 관리자가 그 자리에서 즉시 요청을 승인한다
    Then 그 토큰이 "APPROVED"로 바뀌고, 예비 기기는 곧바로 PIN 화면에 접근할 수 있게 된다

  Scenario: 분실/도난 기기의 신뢰 회수
    Given 이미 "APPROVED" 상태인 토큰을 가진 신뢰 기기가 있다
    And 그 기기를 분실했다는 보고가 들어왔다
    When 관리자가 그 기기의 토큰을 회수(revoke)한다
    Then 그 토큰의 상태가 "REVOKED"로 전환된다
    And 그 기기의 보안 저장소에 토큰 값이 그대로 남아있어도, 서버가 더 이상 유효하다고 인정하지 않으므로 PIN 화면 접근이 즉시 차단된다
    And 이 회수 행위가 감사 로그에 기록된다

  Scenario: 클라이언트가 기기정보를 조작해도 신뢰를 얻을 수 없음
    Given 등록되지 않은 기기가 이미 승인된 다른 기기의 모델명/식별 정보를 그대로 흉내 낸다
    When 이 기기가 PIN 화면 접근을 시도한다
    Then 서버는 그 기기가 "APPROVED" 상태의 유효한 토큰을 제시하지 못하므로 신뢰 기기로 인정하지 않는다
    And "기기 등록 대기 중" 화면만 표시된다

  Scenario: 토큰이 보안 저장소가 아닌 방식으로 노출되면 탈취 위험
    Given 어떤 개발 실수로 토큰이 평문 로그나 비보안 저장소에 남아있다고 가정한다
    When 공격자가 그 값을 그대로 읽어 다른 기기에서 동일 토큰을 제시한다
    Then 서버 입장에서는 진짜 소유 기기와 구분할 수 없어 신뢰 기기로 인정해버린다
    And 그렇기 때문에 클라이언트 구현에서 토큰은 반드시 플랫폼 보안 저장소(Keychain/Keystore)에만 저장해야 한다 (§domain-model.md 2.4-b 요구사항)
```

## Feature 4: 관리자 인증 및 비인가 접근 차단

```gherkin
Feature: 관리자 인증 및 전체 시스템 접근 통제
  관리자는 개인 계정(ID/비밀번호)으로 인증하며,
  인증되지 않은 모든 요청은 종류를 불문하고 차단된다.

  Scenario: 관리자 정상 로그인
    Given 관리자 계정 "admin1"이 사전에 생성되어 있다
    When "admin1"이 올바른 ID/비밀번호로 로그인한다
    Then 로그인이 성공하고 전체 코너/전체 조 조회 및 규칙 변경 권한을 가진 세션이 발급된다

  Scenario: 관리자 비밀번호 오류
    Given 관리자 계정 "admin1"이 존재한다
    When "admin1"이 잘못된 비밀번호로 로그인을 시도한다
    Then 로그인이 거부된다
    And 이 실패 시도가 감사 로그에 기록된다

  Scenario Outline: 로그인 없이 보호된 엔드포인트 접근 시도
    Given 인증되지 않은 클라이언트(세션 없음)가 있다
    When 이 클라이언트가 "<요청>"을 시도한다
    Then 요청이 거부되고 401 인증 필요 응답을 받는다
    And 이 접근 시도가 감사 로그에 기록된다

    Examples:
      | 요청 |
      | 임의 조의 QR 스캔 처리 |
      | 코너별 방문 현황 조회 |
      | 관리자 대시보드 조회 |
      | 코너 목표시간 변경 |

  Scenario: 진행자 권한으로 관리자 전용 기능 접근 시도
    Given "3번 코너 1번 트랙" 진행자로 로그인된 세션이 있다
    When 이 세션으로 "코너 목표시간 변경" API를 호출한다
    Then 요청이 거부된다 (권한 부족)
    And 이 시도가 감사 로그에 기록된다

  Scenario: 모든 인증/인가 이벤트의 감사 로그 적재
    Given 임의의 로그인 성공/실패, 스캔, 규칙 변경, 비인가 접근 시도가 발생했다
    When 관리자가 감사 로그를 조회한다
    Then 각 이벤트에 대해 "행위자(또는 미인증 표시), 시각, 행위 종류, 대상, 성공/실패 여부"가 기록되어 있다
```

## Feature 5: 메시지 — 공지와 트랙 다이렉트 (A10/A11, B6/B7)

```gherkin
Feature: 관리자 ↔ 트랙 메시지
  공지(관리자 → 전체 트랙, 단방향)와 트랙 다이렉트(관리자 ↔ 특정 트랙, 양방향)
  두 채널을 다룬다(§domain-model.md 2.9). 조장/참가자는 앱이 없어 관여하지 않는다.

  Background:
    Given 관리자로 로그인했다

  Scenario: 공지는 발송 시점에 ACTIVE인 트랙에만 노출된다
    Given "3번 코너 1번 트랙"과 "5번 코너 1번 트랙"이 ACTIVE 상태다
    When 관리자가 "15:00부터 간식 시간입니다" 공지를 발송한다
    Then 그 시점 ACTIVE인 두 트랙 모두 공지함(B6)에서 이 공지를 확인할 수 있다

  Scenario: 공지 발송 이후 새로 생성된 트랙은 과거 공지를 보지 못한다
    Given 관리자가 공지를 발송했다
    When 관리자가 그 이후 새 트랙을 생성한다
    Then 그 새 트랙의 공지함에는 발송 이전의 공지가 나타나지 않는다

  Scenario: 공지 읽음 여부는 트랙별로 추적된다
    Given 관리자가 공지 하나를 전체 트랙에 발송했다
    And 일부 트랙만 그 공지를 확인(읽음 처리)했다
    When 관리자가 공지 화면(A10)에서 이 공지의 읽음 현황을 조회한다
    Then 아직 읽지 않은 트랙 수가 표시된다

  Scenario: 트랙 다이렉트 스레드는 트랙 생성과 동시에 빈 상태로 만들어진다
    Given 관리자가 새 트랙을 생성했다
    When 그 트랙의 진행자가 다이렉트 메시지 화면(B7)에 처음 들어간다
    Then 주고받은 메시지가 하나도 없는 빈 스레드가 보인다
    And "아직 나눈 대화가 없습니다" 같은 안내만 표시되고 관리자의 메시지를 기다릴 필요는 없다

  Scenario: 진행자가 먼저 다이렉트 메시지를 보낼 수 있다
    Given "3번 코너 1번 트랙"의 다이렉트 스레드가 비어 있다
    When 그 트랙의 진행자가 "자재가 부족합니다"를 먼저 전송한다
    Then 관리자의 다이렉트 화면(A11)에 이 트랙의 새 메시지가 나타난다
    And 관리자가 먼저 말을 걸어야 한다는 제약은 없다

  Scenario: 진행자는 자신의 트랙 다이렉트 스레드만 접근할 수 있다
    Given "3번 코너 1번 트랙" 진행자로 로그인된 세션이 있다
    When 이 세션으로 "5번 코너 1번 트랙"의 다이렉트 메시지 조회를 시도한다
    Then 요청이 거부된다 (§domain-model.md 2.4 스코프 제한 원칙과 동일)

  Scenario: 트랙 삭제 후에도 다이렉트 스레드 이력은 보존되지만 새 메시지는 보낼 수 없다
    Given "3번 코너 1번 트랙"에 오간 다이렉트 메시지 이력이 있다
    When 관리자가 그 트랙을 삭제한다
    Then 기존 메시지 이력은 사후 조회를 위해 그대로 보존된다
    And 그 트랙으로는 더 이상 새 메시지를 주고받을 수 없다(로그인된 세션이 없으므로)

  Scenario: 트랙 교체로 생성된 새 트랙은 이전 트랙과 다른 새 스레드를 갖는다
    Given "3번 코너 1번 트랙"에 다이렉트 메시지 이력이 있다
    When 관리자가 이 트랙을 "5번 코너"로 교체한다
    Then 새로 생성된 트랙은 비어 있는 새 다이렉트 스레드를 갖는다
    And 이전 트랙의 대화 이력과는 연결되지 않는다

  Scenario: 메시지 발신/수신은 감사 로그에 기록되지 않는다
    Given 관리자와 진행자 사이에 다이렉트 메시지가 여러 건 오갔다
    When 관리자가 감사 로그(A13)를 조회한다
    Then 이 메시지들은 감사 로그 항목으로 나타나지 않는다(보안 이벤트가 아니므로)
```
