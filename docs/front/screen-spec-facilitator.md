# 코너학습 운영 시스템 — 진행자 앱 화면 구성 명세 (모바일)

> 전체 인덱스는 [screen-spec.md](screen-spec.md) 참고 — 관리자 앱 명세는 [screen-spec-admin.md](screen-spec-admin.md).
> §domain-model.md / §scenarios.md / §api-endpoints.md 기반 화면 목록. 시각 스타일(색상/타이포/컴포넌트)은 [design-system.md](../design-system.md) 참고.
> 각 화면은 **목적 / 레이아웃 / 구성 요소 / 상태(States) / 연동 API / 관련 시나리오**로 기술한다.
> 플랫폼: **모바일(iOS/Android, 세로 우선)**.

---

전체 내비게이션 구조: 탭바 없음, 스택 기반. 로그인 이후 사용자는 대부분의 시간을 **B2 메인 트랙 화면**에서 보낸다.

### B0. 기기 등록 대기
- **목적**: 신뢰 토큰이 없는(또는 PENDING/REJECTED) 기기에게 앱 사용 자체를 차단하고 등록 절차를 안내.
- **레이아웃**: 전체화면 안내 — 아이콘 + "기기 등록 대기 중" 제목 + 설명("관리자 승인 후 자동으로 다음 화면으로 이동합니다") + 등록 코드 입력 필드(최초 1회) + SSE/폴링으로 승인 대기 상태 감지.
- **상태**: 코드 입력 전 / PENDING 대기 중(스피너, "승인을 기다리는 중...") / REJECTED(재등록 안내, "새 등록 코드가 필요합니다") / APPROVED 감지 시 자동으로 B1로 전환(사용자 조작 불필요).
- **API**: `POST /device-registrations`
- **시나리오**: scenarios.md Feature 3-b "등록 요청과 동시에 토큰 발급", "관리자의 등록 거절"

### B1. PIN 로그인
- **목적**: 트랙 PIN 6자리 숫자로 트랙 세션 인증.
- **레이아웃**: 중앙 정렬 — "트랙 PIN을 입력하세요" 제목 + OTP 스타일 6박스 표시(design-system §4.6). 실제 입력 요소는 보이지 않는 `<input inputmode="numeric">` 1개뿐이라 OS 기본 숫자 키보드 호출이 한 번으로 끝난다(커스텀 키패드 미구현, 칸별 개별 input도 지양 — 포커스 이동마다 키보드가 깜빡이는 문제 회피).
- **상태**: 기본 / 오류(박스 흔들림 + danger 테두리 + 에러 텍스트 "PIN이 일치하지 않습니다") / 점증형 지연 중(입력 비활성화 + 카운트다운 타이머 "30초 후 다시 시도할 수 있습니다" + "관리자에게 도움 요청" 링크 → B7 다이렉트 스레드로 딥링크).
- **API**: `POST /auth/track/login`
- **시나리오**: scenarios.md Feature 3 "신뢰 기기의 정상 로그인", "잘못된 PIN 입력", "신뢰 기기의 연속 실패 - 점증형 지연"

### B1-b. 코너·트랙 확인 모달
- **목적**: PIN이 서버 검증을 통과한 직후, B2로 곧장 넘어가기 전에 "지금 로그인한 게 맞는 코너·트랙인지" 사람이 한 번 더 확인한다(신규, §domain-model.md 2.4-d). PIN 카드가 잘못 비치되거나 진행자 팀이 카드를 착각해 가져온 경우를 막기 위함 — PIN이 숫자뿐이라 카드만 봐서는 코너를 직관적으로 알 수 없다.
- **레이아웃**: 취소 불가 단일 액션 모달이 아니라 **두 버튼** 모달(design-system §4.4) — 큰 텍스트로 "OO코너 · N번 트랙이 맞습니까?" + "예, 맞습니다"(Primary) / "아니요, 다시 로그인"(Secondary) 두 버튼.
- **인터랙션**: "예" → B2로 진입. "아니요" → 방금 발급된 세션을 폐기하고 B1로 되돌아간다(재인증 유도).
- **상태**: 이 화면은 항상 B1 로그인 성공 직후에만 나타나며 건너뛸 수 없다.
- **API**: 세션은 B1의 `POST /auth/track/login` 응답으로 이미 발급되어 있음(별도 호출 없음). "아니요" 선택 시 그 세션을 폐기하는 엔드포인트 필요(**신규 — api-endpoints.md에 추가 필요**, 예: `POST /auth/track/logout`)
- **시나리오**: scenarios.md Feature 3 "PIN 로그인 직후 코너·트랙 확인", "확인 모달에서 잘못된 배정을 거부하면 재로그인으로 돌아간다"

### B2. 메인 트랙 화면 (홈)
- **목적**: 이 화면 하나로 진행자의 반복 작업(시작/종료)을 전부 처리 — 가장 많이 방문하는 화면.
- **레이아웃**:
  - **상단 헤더(경량)**: 상태 뱃지(IDLE/BUSY, §design-system 1.2) + 우측 아이콘 행(공지함 안읽음 뱃지, 다이렉트 메시지). 코너명·트랙 번호는 헤더에 두지 않는다 — 헤더 폭이 좁아 긴 코너명이 잘리기 쉽고, 상태 뱃지의 점과 나란히 있으면 서로 다른 정보(트랙 상태 vs 연결 상태)가 헷갈렸다(§design-system 0-7 배경).
  - **연결 배너**: 헤더 바로 아래, 평소엔 존재하지 않다가 SSE 재연결 중/끊김일 때만 슬라이드로 등장(§design-system 4.7). 정상 연결 시 아무 UI도 없다.
  - **본문 상단(트랙·코너 정보)**: 작은 대문자 라벨("N번 트랙") 위에 코너명을 큰 타이틀로 배치, 줄바꿈 허용(`text-wrap: balance`) — 트랙은 부가정보, 코너는 주정보라는 위계를 명확히 한다.
  - **본문 중앙(상태+액션)**: IDLE이면 안내 문구 + **"스캔 시작" 버튼**(실제 버튼 크기만 탭 영역 — 카드 전체를 히트박스로 넓히지 않는다, §design-system 0-6). BUSY면 조 번호 + 경과 타이머(목표시간 대비 프로그레스 바, 초과 시 색이 idle→alert로 전환) + **"종료 확인" 버튼**(퇴장, §domain-model 2.6-d — 확정). 카메라를 열지 않고 같은 버튼을 **두 번 탭**한다: 첫 탭은 버튼이 warning 톤 "다시 탭해 확인" 상태로 바뀌며 무장(armed)되고, 제한시간(2~3초) 안에 다시 탭해야 `COMPLETED`로 전환된다. 제한시간 안에 재탭하지 않으면 자동으로 원상태로 복귀(취소에 기록 없음). 모달/바텀시트를 띄우지 않는다 — 롱프레스 방식과 확인 모달 방식은 각각 "현장에서 계속 눌러야 하는 부담"과 "확인창까지 여는 마찰"로 기각했다.
  - **하단 고정**: "수동으로 처리" 버튼(secondary 스타일) 하나만 — 입장 전용(아래 B4 참고), 저빈도 예외 동작이라는 위상에 맞춰 낮춘다.
- **상태**:
  - IDLE: "스캔 시작" 버튼만 활성 탭 영역.
  - BUSY: 경과 타이머 표시, 목표시간 초과 시 색상 변화(하지만 차단하지 않음). "종료 확인" 버튼은 무장 전/무장 중(warning 톤+"다시 탭해 확인") 두 상태를 오간다.
  - **세션 강제 종료 감지(SSE, 트리거 3종 — domain-model.md §2.4)**: 트랙 삭제 / 관리자의 명시적 강제 로그아웃 / **캠프 종료**(신규 명확화 — 이전엔 앞 두 개만 적혀 있었다) 중 무엇이든 감지되면 **BUSY(조 처리 중)여도 예외 없이 즉시** B1로 강제 전환된다. 세 트리거 각각 원인이 다르므로 안내 토스트 문구를 구분한다("이 트랙이 삭제되어…" / "관리자에 의해…" / "코너학습이 종료되어…"). 이때 중단된 방문은 미완료로 남고 별도 복구 절차는 없다.
  - 트랙 교체 감지(SSE `track.replaced`): B8 확인 모달 노출.
- **API**: `GET /tracks/{trackId}/visits/current`, `GET /events/track/{trackId}`(트랙 삭제/강제 로그아웃/캠프 종료 이벤트 포함), `POST /tracks/{trackId}/visits/current/end`(종료 확인 두 번째 탭에서 호출, body 불필요)
- **시나리오**: scenarios.md Feature 1 "정상적인 방문 시작과 종료", "종료 확인은 두 번 탭", "종료 확인 무장 상태를 시간 안에 재탭하지 않으면 자동 취소", "트랙이 이미 사용 중일 때 시작 스캔 거부"(→ 이 화면 자체가 BUSY 상태로 시작 액션을 숨기므로 UI 레벨에서 오조작을 원천 방지), Feature 3 "관리자의 명시적 강제 로그아웃만 세션을 끊는다", "트랙 삭제·캠프 종료로 인한 세션 강제 종료", "BUSY 중에도 세션 강제 종료는 유예 없이 즉시 적용된다"

### B3. QR 스캔 (입장 전용)
- **목적**: 카메라로 조장 QR 배지를 인식해 **시작(입장)** 스캔만 처리한다. 퇴장은 트랙당 진행중 방문이 1개뿐이라 재식별이 필요 없어(§domain-model 2.6-d) 이 화면을 거치지 않고 B2에서 바로 종료 확인으로 처리한다.
- **레이아웃**: 전체화면 카메라 프리뷰 + 중앙 스캔 가이드 프레임 + 상단 "취소"(B2로 복귀) + 하단 "수동 입력으로 전환" 링크.
- **상태**: 스캔 대기 / 인식 성공(초록 프레임 + 체크 애니메이션 + 햅틱, design-system §8) / 인식 실패·거부(카메라 화면 위 바텀시트로 사유 표시 — "이미 완료된 코너입니다" 등, 프레임은 danger 색) / 카메라 권한 거부(안내 화면 + 설정 앱 딥링크).
- **API**: `POST /tracks/{trackId}/visits/start`
- **시나리오**: scenarios.md Feature 1 "이미 완료한 코너 재방문 시도 거부"

### B4. 수동 처리 (조 선택, 입장 전용)
- **목적**: QR 배지 훼손 시 조 목록에서 직접 선택해 **입장**을 기록한다. 퇴장은 배지 상태와 무관하게 항상 B2의 종료 확인으로 처리되므로 이 화면의 대상이 아니다(§domain-model 2.6-b).
- **레이아웃**: 검색창(조 번호 입력) + 조 카드 리스트(design-system §4.5 카드형, 조 번호 대형 표시) + 리스트 상단 "QR 훼손 등으로 인식이 안 되는 경우에만 사용하세요" 안내 배너.
- **구성 요소**: 카드 탭 → 확인 바텀시트("5조를 시작 처리하시겠습니까?") → 확정.
- **상태**: 이미 완료된 조는 리스트에서 비활성 처리(회색, "완료됨" 뱃지) — 오조작 사전 방지.
- **API**: `GET /groups`, `POST /tracks/{trackId}/visits/start` (`method: MANUAL`)
- **시나리오**: scenarios.md Feature 1 "QR 스캔 불능 시 방문 시작과 종료"

### B5. 방문 완료 요약
- **목적**: 종료 확인(B2, 두 번 탭) 직후 소요시간/편차를 짧게 보여주고 자동으로 B2로 복귀.
- **레이아웃**: 전체화면 오버레이 — 큰 체크 아이콘 + "5조 처리 완료" + 소요시간(대형 숫자) + 목표시간 대비 편차(±초, 색상 코딩) + 2~3초 후 자동 닫힘(수동 닫기도 가능).
- **API**: (B2의 `POST /visits/current/end` 응답 데이터 재사용, 별도 호출 없음)

### B6. 공지함
- **목적**: 관리자가 보낸 공지를 확인.
- **레이아웃**: 리스트(최신순, 안읽은 공지는 굵게+좌측 info색 바), 탭 시 전체 내용 펼침, 진입 시 자동 읽음 처리.
- **API**: `GET /messages/broadcast`, `POST /messages/broadcast/{id}/read`
- **시나리오**: domain-model.md §2.9

### B7. 관리자와의 다이렉트 메시지
- **목적**: 자기 트랙과 관리자 간 1:1 대화.
- **레이아웃**: 채팅 스레드 UI(B2 헤더 메시지 아이콘에서 진입) + 하단 입력창 + 입력창 위 정형 빠른 버튼 3개("인원부족" "자재부족" "긴급도움요청" — 탭 한 번으로 정형 메시지 전송, design-system §0-3 원칙대로 급박한 상황에서 타이핑 부담 최소화).
- **상태**: 스레드는 트랙이 생성되는 시점부터 이미 존재하지만 처음엔 비어 있다(domain-model.md §2.9) — **관리자가 먼저 말을 걸어야 진행자가 답할 수 있는 구조가 아니다.** 빈 스레드에는 "아직 나눈 대화가 없습니다 · 도움이 필요하면 먼저 메시지를 보내 보세요" 안내를 보여주고, 진행자가 언제든 먼저 보낼 수 있다(신규 명확화 — 이전 데모는 항상 관리자 메시지로 시작해 반대로 오해할 수 있었다).
- **API**: `GET /tracks/{trackId}/messages`, `POST /tracks/{trackId}/messages`
- **시나리오**: domain-model.md §2.9 "UX 고려"

### B8. 트랙 교체 확인 모달
- **목적**: 관리자가 트랙 교체를 실행했을 때, 접속 중이던 진행자 기기에 통보하고 재PIN 없이 자동 전환.
- **레이아웃**: 취소 불가 단일 버튼 모달(design-system §4.4 세 번째 유형) — "코너가 5번으로 변경되었습니다. 계속하시겠습니까?" + "확인" 버튼 1개.
- **동작**: 확인 시 새 트랙 세션으로 즉시 전환되어 B2(헤더 코너명 갱신) 표시.
- **API**: SSE `track.replaced` 이벤트 수신, 확인 시 세션 교체(§api-endpoints.md "세부 흐름 TBD" 항목 — 확정되면 본 화면의 확인 액션이 호출할 엔드포인트 추가 필요)
- **시나리오**: scenarios.md Feature 2-c "정상적인 트랙 교체"
