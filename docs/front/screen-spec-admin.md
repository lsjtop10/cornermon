# 코너학습 운영 시스템 — 관리자 앱 화면 구성 명세 (iPad)

> 전체 인덱스는 [screen-spec.md](screen-spec.md) 참고 — 진행자 앱 명세는 [screen-spec-facilitator.md](screen-spec-facilitator.md).
> §domain-model.md / §scenarios.md / §api-endpoints.md 기반 화면 목록. 시각 스타일(색상/타이포/컴포넌트)은 [design-system.md](../design-system.md) 참고.
> 각 화면은 **목적 / 레이아웃 / 구성 요소 / 상태(States) / 연동 API / 관련 시나리오**로 기술한다.
> 플랫폼: **iPad(가로 우선)**.

---

전체 내비게이션 구조: 로그인 후 가장 먼저 **캠프 목록(A0-c)**을 거친다. 캠프는 PENDING(준비 중)/ACTIVE(진행 중)/ENDED(종료됨) 세 상태 중 하나이며(§domain-model.md 2.0-a), 어떤 캠프를 여느냐에 따라 사이드바가 완전히 다른 3가지 모드 중 하나로 렌더링된다:

- **운영 모드(ACTIVE)** — 7개 섹션 전체: `대시보드 / 조 현황 / 기기 관리 / 메시지 / 리포트 / 감사 로그 / 설정` (+ 우측 상단 관리자 계정/세션 메뉴). "진행상황 분석" 플로우.
- **준비 모드(PENDING, 신규)** — 4개 섹션만: `코너·트랙 / 조 현황 / 기기 관리 / 설정`. 대시보드·메시지·리포트·감사 로그는 실시간 운영이 전제인 화면이라 아직 시작 전인 캠프에는 없다. "코너·트랙"은 이 모드에서만 사이드바에 나타나는 항목이다(A2B로 연결) — 대시보드가 없어 카드 드릴다운으로 진입할 경로 자체가 없기 때문. "설정 준비" 플로우.
- **리포트 전용 모드(ENDED)** — `리포트 / 감사 로그` 2개 섹션. "사후 분석" 플로우. 감사 로그는 종료된 캠프에 대해서도 조회 가능해야 하므로 포함한다.

세 모드 모두 사이드바 상단에 "← 캠프 목록" 링크가 고정으로 있어 언제든 A0-c로 돌아갈 수 있다. 운영 모드에서는 **"코너·트랙"이 별도 사이드바 항목이 아니다** — 코너 상세(A2)는 대시보드 카드를 눌러 들어가는 드릴다운이고, 여러 코너·트랙을 한꺼번에 다루는 작업은 대시보드에서 진입하는 별도 화면(A2B)으로 뺐다(§design-system.md 3.2 근거). 드릴다운 화면에는 전부 뒤로가기 버튼을 둔다(§design-system.md 0-8).

### A0. 로그인
- **목적**: 관리자 ID/비밀번호 인증.
- **레이아웃**: 중앙 정렬 카드(폭 400pt), 로고/캠프명 상단, ID·비밀번호 입력, "로그인" Primary 버튼. 사이드바 없음(§design-system 3.2 전체화면 예외).
- **구성 요소**: 입력 필드 2개, 에러 텍스트 영역(비밀번호 오류 시 인라인 표시, danger 색), "로그인 상태 유지" 안내 문구(리프레시 토큰이 자동 처리하므로 별도 체크박스 불필요 — §technical-design.md 2.2-b).
- **상태**: 기본 / 입력 중 / 검증 실패(흔들림 애니메이션 + 에러 텍스트) / 로딩.
- **후속 분기(변경)**: 로그인 성공 시 캠프 개수와 무관하게 **항상 A0-c 캠프 목록으로 이동한다**. 기존에는 캠프가 하나도 없는 최초 실행 시 A0-b 초기 설정 마법사로 자동 분기했으나, 이 분기를 없앴다 — 로그인 직후 목적지가 캠프 유무에 따라 달라지면 예측하기 어렵고, "캠프 목록"이라는 단일 진입점을 두는 편이 내비게이션 구조상 더 단순하다. 캠프가 0개인 경우는 A0-c가 empty state로 표현하고, 새 캠프 생성은 그 화면의 "+ 새 캠프 시작" 버튼을 통해서만 A0-b로 진입한다. **A1 대시보드로 곧장 가지 않는다** — 여러 캠프를 오가야 하므로 목록이 항상 먼저다(§domain-model.md 2.0-a).
- **API**: `POST /auth/admin/login`, `GET /camps`(캠프 존재 여부·목록 확인)
- **시나리오**: scenarios.md Feature 4 "관리자 정상 로그인", "관리자 비밀번호 오류"

### A0-c. 캠프 목록
- **목적**: 진행 중·준비 중·종료된 캠프를 한곳에서 이어가거나, 새 캠프를 시작할 지점을 하나로 모은다(신규). 이전에는 로그인하면 곧장 대시보드(A1)로 들어가서 "코너학습 종료 후 리포트는 봤는데 새 캠프는 어떻게 시작하지"가 막다른 골목이었다 — 이 화면이 그 빈틈을 메운다.
- **레이아웃**: 사이드바 없는 전체화면(§design-system 3.2 전체화면 예외) + 상단 "QR 배지 관리"(A0-d)·"+ 새 캠프 시작" 버튼 + 캠프 카드 목록(진행 중 → 준비 중 → 종료됨 순 — 상태별 배지, 기간, 요약).
- **인터랙션**: **진행 중** 카드 클릭 → 운영 사이드바 전체로 A1 진입("진행상황 분석"). **준비 중** 카드 클릭(신규) → 준비 모드 축소 사이드바로 A2B 진입("설정 준비" — 대시보드가 없어 코너·트랙 화면이 기본 진입점). **종료됨** 카드 클릭 → 리포트 전용 축소 사이드바로 A12 진입("사후 분석", 곧장 실제 리포트 표시). "+ 새 캠프 시작" → A0-b 초기 설정 마법사(완료 시 새 캠프가 이 목록에 **"준비 중"**으로 추가되며 곧장 그 캠프의 준비 모드로 이동 — 진행 중이 아니다).
- **캠프 상태 판단 기준(신규, 중요)**: PENDING→ACTIVE 전이는 코너·트랙이 갖춰졌는지가 아니라, 관리자가 그 캠프의 준비 모드 안에서 **"코너학습 시작"을 명시적으로 확정**했는지로만 결정된다(A0-e, §domain-model.md 2.0-a). 코너가 10개 다 준비돼 있어도 시작을 누르기 전까지는 "준비 중" 배지 그대로다.
- **상태**: 캠프가 하나도 없으면(변경 — 이제 최초 실행 시에도 로그인 후 항상 이 화면에 도달한다) empty state("아직 캠프가 없습니다" 안내 + "+ 새 캠프 시작" CTA 강조).
- **API**: `GET /camps?status=`(PENDING/ACTIVE/ENDED 모두 반환)
- **시나리오**: scenarios.md Feature 2-g 전체, Feature 2-i "준비 중 캠프와 명시적 시작"

### A0-e. 코너학습 시작
- **목적**: PENDING 캠프를 ACTIVE로 전이시키는 명시적 확정 조작(신규) — "코너학습 종료"(A14)의 대응 짝. 코너·트랙이 있다고 자동으로 시작되지 않게 해서, 준비 중에 실수로 진행자 로그인이 열리는 일을 막는다.
- **레이아웃**: 준비 모드 상단 바 우측 고정 "코너학습 시작" Primary 버튼(A14의 "코너학습 종료" destructive 버튼과 자리는 같지만 준비 모드에서만 보임, 운영 모드에서는 반대로 숨겨짐) → 확인 모달("PIN 카드는 이미 발급돼 있으니 시작 전까지는 로그인이 거부됩니다" 안내 + "시작 확정" 버튼).
- **후속 동작**: 확정 즉시 캠프 상태가 PENDING→ACTIVE로 바뀌고, 사이드바가 준비 모드(4개)에서 운영 모드(7개) 전체로 전환되며 대시보드(A1)로 이동한다. 이 순간부터 그 캠프에 속한 트랙의 PIN 로그인이 실제로 통하기 시작한다(§domain-model.md 2.4-e).
- **API**: `POST /camps/{id}/start`
- **시나리오**: scenarios.md Feature 2-i 전체

### A0-d. QR 배지 사전 생성
- **목적**: 조(Group)와 그 QR 배지를 **특정 캠프와 무관하게** 미리 대량으로 준비한다(신규, domain-model.md §2.1-a). 현장에서 급하게 QR을 인쇄해 스코어판에 붙일 시간이 없다는 제약 때문에 "조를 먼저 만들고 QR을 내보내는" 순서를 뒤집었다 — **배지가 먼저, 조 등록은 나중**이다. 이 화면은 초기 설정 마법사(A0-b)의 일부가 아니다 — 코너·트랙 준비와는 완전히 다른 시점(그보다 훨씬 이전, 캠프를 정하기 전에도 가능)에 이뤄지는 별개의 준비 작업이라 캠프 목록(A0-c)에서 캠프 선택과 무관하게 접근한다.
- **레이아웃**: 사이드바 없는 전체화면(§design-system 3.2 전체화면 예외), 캠프 목록(A0-c)에서 뒤로가기로 진입. 생성 수량 입력(기본 40) + "배지 생성" 버튼 + "스티커 PDF로 내보내기" 버튼 + 미배정/배정됨 카운트 + 배지 테이블(배지 ID, 상태, 등록된 조).
- **인터랙션**: "배지 생성" → 입력한 수량만큼 미배정(UNASSIGNED) 배지를 즉시 만든다. "스티커 PDF로 내보내기" → 현재 미배정 배지 전체를 스티커 인쇄용 PDF로 내보낸다. **iPad에서 직접 인쇄(AirPrint)하지 않는다** — 인쇄는 이 PDF를 받아 다른 컴퓨터에서 한다(§A12 리포트와 동일한 결정, 브라우저/OS별 인쇄 드라이버 차이에 얽매이지 않기 위함).
- **상태**: 생성 직후 카운트·테이블에 즉시 반영. 여유분(예상 조 수보다 넉넉히, 예: 20개 조 예상 시 40장)을 두는 이유는 당일 조 수가 달라지거나 배지가 훼손돼도 추가 인쇄 없이 대응하기 위함 — 스티커 형태로 제작하면 내구성이 있어 이번 캠프에서 다 못 쓴 미배정 배지는 다음 캠프에서도 재사용한다.
- **API**: `POST /badges/bulk-generate`, `GET /badges/export`(스티커 PDF), `GET /badges?status=`
- **시나리오**: scenarios.md Feature 2-h 전체

### A0-b. 초기 설정 마법사
- **목적**: 캠프 정보와 코너·트랙을 행사 시작 전 한 번에 준비한다. 기존에는 코너/트랙 생성 수단이 A2 안의 "트랙 추가" 버튼 하나뿐이라 코너 10개를 하나하나 만들어야 하는 입력 노가다였다 — 이 화면이 그 문제를 해결한다. **이 마법사는 캠프·코너·트랙만 다룬다 — 조(Group)와 QR 배지는 여기서 생성하지 않는다**(신규 명확화). 조는 배지 등록(A5의 "+ 조 등록")을 통해서만, 필요할 때 하나씩 생성된다 — 배지 자체는 이 마법사보다 훨씬 전에 A0-d에서 미리 만들어둔다.
- **레이아웃**: 사이드바 없는 전체화면(§design-system 3.2), 중앙 640pt 폭 카드. 상단에 3단계 스텝 인디케이터(§design-system 4.6 단계형 마법사), 하단에 이전/다음 고정 버튼.
  - **1단계 — 캠프 정보**: 캠프 이름, 시작일/종료일 입력(domain-model.md §2.0의 Camp 속성 그대로).
  - **2단계 — 코너·트랙 일괄 생성**: 코너 이름을 줄바꿈으로 구분해 붙여넣는 텍스트 영역 + "예시 10개로 빠르게 시작" 템플릿 버튼 + 기본 목표시간/코너당 기본 트랙 수 입력. 파싱된 결과는 표로 미리보기되며, 코너별로 이름·목표시간·트랙 수를 개별 조정하거나 행을 삭제할 수 있다.
  - **3단계 — 검토**: 캠프 정보·코너 수·생성될 트랙 총수를 요약하고, "설정 완료 → 코너·트랙 준비로" 확정 시 모든 트랙에 PIN이 자동 발급된다. **곧장 ACTIVE가 되지 않는다(신규)** — 새 캠프는 "준비 중(PENDING)" 상태로 캠프 목록에 추가되고, 준비 모드(A2B)로 이동한다. 진행자 PIN 로그인은 관리자가 별도로 "코너학습 시작"(A0-e)을 확정해야 열린다.
- **상태**: 2단계에서 코너가 0개인 채로 다음을 누르면 진행 차단("코너를 1개 이상 추가하세요") — 완전히 빈 캠프로 시작하는 걸 막는 소프트 가드.
- **API**: `POST /camps`(PENDING 상태로 생성), `POST /corners`(배열 입력으로 코너+트랙 일괄 생성)
- **시나리오**: scenarios.md Feature 2-d 전체

### A1. 대시보드 (홈)
- **목적**: 10개 코너 상태를 한눈에(스캔) 파악, 병목 조기 감지.
- **레이아웃**: 상단 요약 바(전체 완주율, 진행중 조 수, 프로그램 경과시간, **안읽은 다이렉트 메시지 수 — 탭하면 A11로 이동**, 수정) + 그 아래 정렬 드롭다운(좌측)과 "트랙 일괄 관리 →" 링크(우측, A2B 진입점) + 코너 상태 카드 그리드(3~4열). 실시간 연결 상태는 상시 인디케이터 없이, 끊겼을 때만 헤더 아래 배너로 등장한다(§design-system 4.7 — 진행자 앱과 동일 원칙을 관리자 앱에도 적용).
- **구성 요소**: 상태 카드(design-system §4.1) 10장, 상단 필터 칩(전체/BUSY/IDLE/미가동/병목만 보기), 우측 상단 "공지 발송" 퀵 액션 버튼.
- **정렬(신규)**: 드롭다운 4가지 — 코너 번호순(기본) / 이름순(가나다) / 평균편차 높은순(병목 후보를 위로 몰아 훑어보기) / 평균편차 낮은순(§design-system 4.5-b 카드 그리드 정렬 패턴). 미가동 코너(편차 데이터 없음)는 정렬 방향과 무관하게 항상 맨 뒤로 밀린다.
- **코너 카드 색상**: 미가동(회색)/유휴(진한 중립색)/정상(초록) 3색만 쓴다(§design-system 1.2-b) — 트랙 일부만 가동 중인지 전부 가동 중인지는 색으로 구분하지 않는다. 병목 판정을 받은 코너는 이 색과 무관하게 좌측 보더가 항상 빨간색으로 덮어써진다.
- **편차 표시**: 카드에 "평균편차"를 별도 줄로 두지 않는다. "평균 10:40 **(+2:30)** · 최근 10건"처럼 **평균 소요시간 옆 괄호에 편차를 붙이고**, 표본 수(최근 N건)를 함께 표기해 "직전 1건이 아니라 평균"임을 분명히 한다 — 관리자가 1차로 궁금한 건 "이 코너가 보통 몇 분 걸리는가"이고 목표 대비 편차는 그 부연 정보라서 순서를 이렇게 잡았다.
- **병목 판정 기준은 여기 없다**: 자주 바꾸는 값이 아니라서 설정(A15)으로 뺐다 — 대시보드는 "지금 상황을 보는" 화면이지 설정 화면이 아니다(초안에서는 인라인 링크+모달로 뒀다가 이 원칙에 따라 옮겼다).
- **인터랙션**: 카드 탭 → 코너 상세(A2)로 이동(뒤로가기로 복귀).
- **상태**: 정상 / 데이터 로딩 스켈레톤. **코너학습 종료 후에는 이 화면이 아니라 캠프 목록(A0-c)으로 돌아간다**(A14 참고) — "읽기 전용 대시보드"라는 제3의 상태를 따로 두지 않는다.
- **API**: `GET /events/admin`(SSE 스냅샷), `GET /reports/live-summary`
- **시나리오**: scenarios.md Feature 2 "전체 트랙 가동 상태 감지", "유휴 코너 감지"

### A15. 설정
- **목적**: 자주 바꾸지 않는 캠프 단위 설정값을 모아두는 화면. 자주 바꾸는 값(코너 목표시간 등)은 각 화면에 그대로 두고, "가끔·의도적으로만 바꾸는" 값만 여기로 모은다.
- **레이아웃**: 사이드바 최상위 항목(드릴다운 아님, 뒤로가기 불필요) — ① 캠프 정보(이름, 시작일/종료일, 저장 버튼) ② 병목 판정 기준(§domain-model.md 2.6-e — 최소 표본 건수, 목표시간 대비 편차 비율, 저장 버튼).
- **상태**: 저장 성공 시 토스트. 병목 기준을 저장하면 대시보드(A1)의 병목 판정에 즉시 반영된다(재조회 불필요).
- **API**: `GET /camps/{id}`, `PATCH /camps/{id}`(이름/기간, 병목 기준 파라미터 모두 이 엔드포인트로 저장)
- **시나리오**: scenarios.md Feature 2-f 전체

### A2. 코너 상세
- **목적**: 특정 코너의 트랙 목록과 목표시간 규칙을 관리.
- **레이아웃**: 제목 왼쪽에 뒤로가기 버튼(→A1, §design-system 0-8) + 상단 코너 요약(이름, 상태, 목표시간 — 인라인 편집 가능) + 트랙 목록(테이블: 트랙 번호, 상태, 현재 조, PIN(마스킹), 액션).
- **구성 요소**: "트랙 추가" Primary 버튼, 트랙 행마다 **"PIN 보기"(그 트랙 하나만)** + "교체" + "삭제" 액션, 목표시간 편차 미니 차트(최근 처리된 방문들의 편차 스파크라인). "PIN 보기"는 서버가 반환한 PIN을 팝업으로 표시하고 **복사하기**와 **PDF 내보내기**를 제공한다. **"전체 PIN 내보내기"는 이 화면에 없다** — 특정 코너 하나의 맥락에 "전체"라는 범위가 어울리지 않고 찾기도 어렵다는 이유로 A2B(트랙 일괄 관리)로 옮겼다(§A4 참고).
- **상태**: 트랙 0개(미가동, empty state + "트랙 추가" CTA 강조) / 트랙 삭제 진행 중(로딩 인디케이터) / 삭제 하드 블록(§A2-모달1) / 삭제 소프트 확인(§A2-모달2).
- **모달**:
  - A2-모달1 (하드 블록): "진행 중인 방문이 있어 삭제할 수 없습니다." (design-system §4.4)
  - A2-모달2 (소프트 확인): "이 코너를 서비스할 트랙이 없어집니다. 계속하시겠습니까?"
  - A2-모달3 (목표시간 변경 확인): 변경 전/후 값 비교 표시 후 저장.
- **API**: `GET /corners`, `PATCH /corners/{id}`, `GET /tracks`, `POST /corners/{cornerId}/tracks`, `DELETE /tracks/{id}`, `POST /tracks/{id}/regenerate-pin`, `GET /tracks/{id}/export`(트랙 단건 PIN)
- **시나리오**: scenarios.md Feature 2-b 전체, Feature 2 "병목 코너에 대한 규칙 변경", "두 관리자의 동시 규칙 변경 충돌"(마지막 저장 시각 표시로 last-write-wins를 투명하게 노출)

### A2B. 트랙 일괄 관리
- **목적**: 여러 코너에 걸친 트랙을 한 화면에서 골라 목표시간을 일괄 변경하거나 한꺼번에 삭제한다. 운영 모드(ACTIVE 캠프)에서는 사이드바에 "코너·트랙" 항목을 없앤 대신 마련한 진입점(대시보드 A1 우측 상단 "트랙 일괄 관리 →"). **준비 모드(PENDING 캠프)에서는 이 화면이 사이드바 "코너·트랙" 항목 자체이자 기본 진입 화면이다(신규)** — 준비 모드엔 대시보드가 없어 카드로 드릴다운할 경로가 없기 때문에, 코너·트랙 편집이 여기서 항상 직접 접근 가능해야 한다.
- **레이아웃**: 제목 왼쪽 뒤로가기(운영 모드에서는 →A1, 준비 모드에서는 뒤로가기 대신 사이드바 "코너·트랙" 자체가 진입점이라 생략 가능) + 상단 액션 바(선택 개수, 상태 필터 드롭다운, 목표시간 일괄 변경 입력+적용 버튼, "선택 삭제" 버튼) + 전체 트랙 테이블(체크박스, 코너/트랙/상태 정렬 가능한 컬럼 헤더, 현재 조, PIN).
- **필터·정렬**: 상태(전체/IDLE만/BUSY만) 드롭다운 + 코너·트랙 번호·상태 컬럼 클릭 정렬(§design-system 4.5-b).
- **상태**: 선택 항목 중 `IN_PROGRESS`(BUSY) 트랙이 하나라도 있으면 "선택 삭제" 버튼이 비활성화되고 경고 문구가 뜬다 — **부분 삭제로 어중간한 상태를 만들지 않고 전체를 거부**한다(§domain-model.md 삭제 가드①을 집합 단위로 승격 적용). "전체 선택" 체크박스는 **현재 필터로 화면에 보이는 트랙만** 선택한다 — 필터 밖의 트랙까지 조용히 선택되면 되돌리기 어려운 일괄 삭제의 오조작 위험이 커진다(§design-system 4.5-b).
- **API**: `GET /tracks?status=&sort=&order=`, `PATCH /corners`(일괄 목표시간 변경), `POST /tracks/bulk-delete`(일괄 삭제)
- **시나리오**: scenarios.md Feature 2-e 전체

### A3. 트랙 교체
- **목적**: "코너가 바뀐다"는 요구를 트랙 삭제+재생성으로 원스텝 처리.
- **레이아웃**: 모달(폭 480pt) — 현재 코너/트랙 정보 → 화살표 → 신규 코너 선택 드롭다운 → 영향 요약("PIN 카드 재인쇄 필요", "접속 중인 기기 1대는 자동 재인증됩니다") → 실행 버튼.
- **상태**: 진행 중 방문 있음(하드 블록) / 마지막 트랙(소프트 확인) / 정상 진행.
- **API**: `POST /tracks/{id}/replace`
- **시나리오**: scenarios.md Feature 2-c 전체

### A4. PIN 카드 내보내기 (전체)
- **목적**: 현재 ACTIVE 트랙 전체의 PIN 목록을 CSV로 내보내 인쇄. 트랙 하나만 필요할 때는 A2 트랙 행의 개별 "PIN 보기" 버튼을 쓴다(전체를 다시 뽑을 필요 없도록 분리).
- **레이아웃**: **A2B(트랙 일괄 관리) 상단**의 "전체 PIN CSV 다운로드" 버튼 → 서버 JSON 응답에서 UTF-8 BOM CSV 파일을 생성해 다운로드/공유 시트를 연 뒤 완료 토스트. 처음엔 A2(코너 상세) 헤더에 있었으나 "특정 코너 하나의 맥락에 '전체' 범위 버튼이 있으면 찾기 어렵다"는 지적으로 이미 코너를 가로지르는 화면인 A2B로 옮겼다.
- **구성 요소**: 최근 내보내기 이력(시각, 내보낸 관리자) 표시 — 인쇄본이 최신인지 판단 근거.
- **API**: `GET /tracks/export`(JSON 응답을 클라이언트에서 CSV로 생성한 뒤 `share_plus`로 공유)
- **시나리오**: scenarios.md Feature 2-b "현재 ACTIVE 트랙 PIN 목록 엑셀 내보내기"

### A5. 조 현황 목록
- **목적**: 조의 순회표/완주 상태를 조회하고, **QR 배지를 등록해 새 조를 만드는 지점**(신규)이기도 하다 — 조를 이름만으로 미리 만들어두는 별도 화면은 없다(domain-model.md §2.1-a).
- **레이아웃**: 필터 칩(전체/완주/부분완주) + 우측 상단 "+ 조 등록" 버튼(신규) + 테이블(조 번호, 현재 상태, 완료 코너 수 "7/10", 마지막 스캔 시각) — 조/상태/완료 코너 수 컬럼은 클릭 정렬 가능(§design-system 4.5-b).
- **구성 요소**: 행 탭 → 조 상세(A6) 이동. 완주 조는 초록 체크 뱃지. 필터 우측에 "N / 전체건" 카운트 표기.
- **"+ 조 등록" 모달(신규)**: 탭 2개 — **카메라로 스캔**(현장에서 스코어판에 붙은 배지를 촬영, 인식되면 조 이름 입력란이 나타남) / **목록에서 선택**(미배정 배지를 짧은 ID로 검색·선택, 사무실에서도 가능). 두 경로 모두 조 이름을 입력해야 "등록 확정"이 활성화된다. 확정 즉시 조가 생성되고 이 목록에 나타난다. 이미 이번 캠프에서 배정된 배지를 다시 선택/스캔하면 등록이 거부된다(domain-model.md §2.1-a).
- **API**: `GET /groups?filter=&sort=&order=`, `POST /badges/{id}/register`(목록 탭), `POST /badges/scan-register`(스캔 탭), `GET /badges?status=&search=`(목록 탭 검색)
- **시나리오**: scenarios.md Feature 2-h 전체, analytics-model §1.4 조 레벨 지표와 연결

### A6. 조 상세 (순회표)
- **목적**: 특정 조의 10개 코너 완료 현황과 방문 이력을 확인.
- **레이아웃**: 제목 왼쪽 뒤로가기(→A5, §design-system 0-8) + 상단 조 요약 + 10개 코너 그리드(완료/미완료 아이콘) + 방문 이력 테이블(코너, 트랙, 시작/종료 시각, 소요시간, 편차, 입력방식 뱃지).
- **구성 요소**: 미완료 코너에 "중복방문 예외 승인" 액션 진입점(코너가 이미 COMPLETED인데 재방문이 필요한 특수 케이스용 — A7 모달 호출).
- **API**: `GET /groups/{id}`, `GET /groups/{id}/visits`
- **시나리오**: scenarios.md Feature 1 "프로그램 종료 시각까지 일부 코너를 방문하지 못한 조의 처리"

### A7. 중복 방문 예외 승인 (모달)
- **목적**: 이미 COMPLETED된 조-코너 조합의 재방문을 관리자가 예외 승인.
- **레이아웃**: 모달 — 대상 조/코너 표시, 승인 사유 텍스트 입력(선택), "예외 승인" destructive-tone 버튼(일반적이지 않은 조작이므로 강조 색 사용, 삭제는 아니지만 경고 톤).
- **API**: `POST /visits/exception-approve`
- **시나리오**: scenarios.md Feature 1 "관리자 승인을 통한 예외적 재방문 허용"

### A8. 기기 등록 관리
- **목적**: 진행자 기기의 신뢰 등록 요청을 승인/거절, 기존 신뢰 기기 회수.
- **레이아웃**: 상단 탭(대기 중 PENDING / 승인됨 APPROVED / 거절·회수 이력) + 리스트(요청 시각, 기기 식별 힌트, 액션 버튼).
- **구성 요소**: PENDING 행에 "승인"(primary)/"거절"(secondary) 버튼, APPROVED 행에 "회수"(destructive) 버튼 + 회수 확인 모달("분실/도난 대응 시 사용하세요" 안내 문구).
- **상태**: 대기 목록 비어있음(empty state) / 실시간 신규 요청 도착 시 리스트 상단에 하이라이트 애니메이션 + 배지 카운트.
- **API**: `GET /device-registrations`, `POST /device-registrations/{id}/approve|reject|revoke`
- **시나리오**: scenarios.md Feature 3-b 전체

### A9. PIN 잠금 해제 / 세션 관리
- **목적**: 신뢰 기기의 점증형 지연(잠금) 즉시 해제, 진행자 트랙 세션 강제 로그아웃, 관리자 자신/공동 관리자 세션 회수.
- **레이아웃**: 3개 섹션 카드 — ① 현재 잠금된 기기 목록(트랙명, 실패 횟수, 남은 지연시간, "잠금 해제" 버튼) ② 활성 진행자 세션 목록(트랙별, "강제 로그아웃" 버튼) ③ 관리자 세션 목록(기기, 마지막 활동, "세션 종료" 버튼).
- **API**: `POST /auth/track/lockout/{deviceId}/release`, `POST /auth/track/{trackId}/force-logout`, `GET /auth/admin/sessions`, `POST /auth/admin/sessions/{id}/revoke`
- **시나리오**: scenarios.md Feature 3 "관리자의 즉시 잠금 해제", "관리자의 명시적 강제 로그아웃만 세션을 끊는다"

사이드바엔 "메시지" 항목이 하나뿐이지만 실제로는 공지(A10)·다이렉트(A11) 두 화면이다 — A12(리포트)와 같은 탭바를 두 화면 상단에 둬서 오갈 수 있게 한다(수정 — 이전엔 A11이 사이드바 어디에서도 갈 수 없는 리뷰용 바로가기로만 존재해 실제 앱 흐름에서 도달 불가능했다). 대시보드(A1)의 "안읽은 다이렉트" 요약 카드를 탭해도 곧장 A11로 온다.

### A10. 메시지 — 공지 채널
- **목적**: 전체 ACTIVE 트랙에 단방향 공지 발송, 읽음 현황 추적.
- **레이아웃**: 상단 탭바(공지/다이렉트, 신규) + 좌측 발송 이력 리스트(최신순) + 우측 상단 "새 공지 작성" 버튼(모달, 텍스트 입력 + 발송) + 선택한 공지의 읽음 현황(트랙별 읽음/안읽음 그리드, 안읽음 트랙 강조).
- **API**: `POST /messages/broadcast`, `GET /messages/broadcast`, `GET /messages/broadcast/{id}/receipts`
- **시나리오**: domain-model.md §2.9, analytics-model.md §1.6 "공지별 읽음 도달률"

### A11. 메시지 — 트랙 다이렉트
- **목적**: 특정 트랙과의 1:1 대화.
- **레이아웃**: 상단 탭바(공지/다이렉트, 신규 — 다이렉트 탭에 안읽음 총합 뱃지 표시) + 좌측 트랙 목록(최근 메시지 미리보기, 안읽음 카운트 뱃지) + 우측 채팅 스레드(말풍선 UI, 관리자 발신은 우측 정렬 브랜드색, 트랙 발신은 좌측 정렬 뉴트럴) + 하단 입력창.
- **구성 요소**: 트랙 쪽에서 온 정형 빠른 버튼 메시지("인원부족"/"자재부족"/"긴급도움요청")는 danger 톤 태그로 강조 표시. **스레드는 트랙 생성 시점부터 비어 있는 채로 존재한다** — 관리자가 먼저 말을 걸어야 하는 구조가 아니라 트랙(진행자) 쪽이 먼저 보낸 스레드도 동일하게 취급된다(domain-model.md §2.9, screen-spec-facilitator.md B7과 일관).
- **API**: `GET /tracks/{trackId}/messages`, `POST /tracks/{trackId}/messages`
- **시나리오**: domain-model.md §2.9

### A12. 리포트 (사후 분석)
- **목적**: 코너학습 종료 후 집계 리포트 열람 및 내보내기. 두 경로로 들어올 수 있다 — ① 아직 진행 중인 캠프의 운영 사이드바에서 "리포트" 클릭(종료 전이면 empty state), ② 캠프 목록(A0-c)에서 종료된 캠프를 클릭(리포트 전용 축소 사이드바로 진입, 곧장 실제 데이터 표시).
- **레이아웃**: 상단 탭 3개 — **요약 / 코너별 / 조별**로 세분화(기존엔 요약 화면 하나에 다 욱여넣었다).
  - **요약**: 캠프 레벨 요약 카드(완주율, 평균편차, 수동 처리 비율) + 코너별 병목 랭킹 Top3.
  - **코너별**: 코너마다 1행 — 처리 완료 조 수, **평균 소요시간(편차)**(A1 대시보드와 동일하게 "10:40 (+2:30)" 형식으로 병합 표기), 편차>0 비율(analytics-model §1.2).
  - **조별**: 조마다 1행 — 완주 여부, 완료 코너 수, 총 활동시간, 미완료 코너 목록(analytics-model §1.4).
  - 시계열/운영지표는 이번 세분화 범위 밖 — 필요해지면 탭을 추가한다(§4 확인 필요 사항).
  - 탭 우측 "PDF로 내보내기"(신규) — iPad에서 직접 인쇄(AirPrint)하지 않고 PDF만 내보낸다. 인쇄는 그 PDF를 받아 다른 컴퓨터에서 하는 방식으로 정했다(§A0-d와 동일한 결정).
- **상태**: 코너학습 진행 중(리포트 미생성, "코너학습 종료 후 이용 가능" 안내) / 생성됨. 한 번 생성된 리포트는 종료 후 데이터가 더 이상 바뀌지 않으므로 재생성 상태는 없다.
- **API**: `GET /reports/current`(탭 3개가 이 응답 하나를 구간별로 렌더링), `POST /reports/generate`, `GET /reports/current/export`(PDF)
- **시나리오**: analytics-model.md §1 전체, §3

### A13. 감사 로그
- **목적**: 모든 인증/스캔/규칙변경/관리자 조작 이력을 필터링해 조회.
- **레이아웃**: 상단 필터 바(행위자 자유 텍스트 검색, 행위 종류 드롭다운, 결과(성공/실패) 드롭다운, 활성 필터 있으면 "필터 초기화 (N)" 버튼, 우측에 "N / 전체건" 카운트) + 테이블(시각/행위자/행위 종류 컬럼 클릭 정렬, 대상, 결과 성공/실패 뱃지) — §design-system 4.5-b 필터 바 패턴.
- **구성 요소**: 실패 행은 danger 톤 좌측 바로 강조. 정렬 컬럼은 헤더에 ▲▼ 아이콘(비활성은 ↕)으로 방향 표시.
  > **각주(구현 시 정정)**: `GET /audit-logs`에는 `sort`/`order` 파라미터가 없다 — 서버 정렬 미지원으로 컬럼 정렬 UI(▲▼/↕ 아이콘, 클릭 정렬) 자체를 제외했다. 우측 카운트도 "N / 전체건"이 아니라 "현재까지 N건 로드됨"으로 대체했다(서버가 커서 페이지네이션이라 전체 건수를 안 준다). 근거: `11_a13_a14_a15_audit_end_settings.md` §2.2, §2.3 각주.
- **API**: `GET /audit-logs?actor=&action=&result=&sort=&order=`
- **시나리오**: scenarios.md Feature 4 "모든 인증/인가 이벤트의 감사 로그 적재", 그리고 각 Feature의 "감사 로그에 기록된다" 문구가 붙은 모든 시나리오.

### A14. 코너학습 종료
- **목적**: 프로그램(코너학습) 수동 종료 선언.
- **레이아웃**: 상단 바 우측 고정 "코너학습 종료" destructive 버튼(대시보드뿐 아니라 앱 어디서든 접근 가능) → 확인 모달(현재 완주/부분완주 조 수 요약 표시 후 "종료 선언" destructive 버튼). **라벨을 "캠프 종료"가 아니라 "코너학습 종료"로 쓴다** — 캠프는 여러 프로그램을 아우르는 상위 개념이고(domain-model.md §2.0), 이 조작이 끝내는 건 캠프 자체가 아니라 그 안의 코너학습 프로그램 1회차이기 때문에 관리자가 오해하지 않도록 명확히 구분했다.
- **후속 동작**: 종료 즉시 `POST /reports/generate` 자동 트리거. **A12(리포트)로 유도하지 않는다** — 그 캠프의 대시보드에 머무르는 대신 캠프 목록(A0-c)으로 돌아간다(신규, §domain-model.md 2.0-a). 방금 끝난 캠프는 그 목록에서 "종료됨" 배지로 나타나며, 클릭하면 리포트 전용 화면(A12)으로 들어가거나 "+ 새 캠프 시작"으로 바로 다음 코너학습을 준비할 수 있다.
- **API**: `POST /camps/{id}/end`
- **시나리오**: scenarios.md Feature 1 "프로그램 종료 시각까지 일부 코너를 방문하지 못한 조의 처리", Feature 2-g "코너학습 종료 후 캠프 목록으로 복귀"

## 관리자 앱에 남은 확인 필요 사항

- 관리자 iPad 세로 모드(portrait) 사용 시 사이드바 축소 동작의 상세 인터랙션은 별도 정의 필요(현재는 "지원하되 최적화 대상 아님"으로만 명시, design-system.md §3.2).
- A0-b·A2B·A15의 Given-When-Then 시나리오는 scenarios.md Feature 2-d/2-e/2-f로 추가 완료. 다만 **코너 이름 중복 시 처리**는 아직 시나리오화하지 않았다 — `POST /corners` 부분 실패 트랜잭션 범위(아래 항목)가 정해지면 함께 정리할 것.
- A0-c(캠프 목록)의 Given-When-Then 시나리오는 scenarios.md Feature 2-g로 추가 완료.
- **캠프 컨텍스트 전달 방식**은 api-endpoints.md "아직 세부화 안 된 지점"에 미정으로 남겨뒀다 — URL에 `campId`를 명시할지, 세션의 "현재 선택된 캠프"로 암묵 처리할지에 따라 A2/A2B/A5/A6/A12/A13 등 캠프 하위 화면의 API 호출부가 달라질 수 있다.
- **A0-d(QR 배지 사전 생성)와 A5의 "+ 조 등록"**은 scenarios.md Feature 2-h로 추가 완료(domain-model.md §2.1-a). 다만 다음은 아직 미정이다 — api-endpoints.md "아직 세부화 안 된 지점" 참고: ① 캠프가 끝난 뒤 그 캠프에서 배정됐던 배지를 언제·어떻게 미배정으로 되돌리는지(서버 자동 vs 데이터 모델 이중화), ② 등록된 배지가 현장에서 분실·훼손됐을 때 조는 유지한 채 배지만 교체하는 액션의 유무.
- **캠프 상태 PENDING→ACTIVE 전이는 결정 완료**: "코너·트랙이 준비되면"이라는 암묵적 기준을 버리고, 관리자가 준비 모드에서 "코너학습 시작"(A0-e, `POST /camps/{id}/start`)을 명시적으로 확정해야만 전이된다(domain-model.md §2.0-a, §2.4-e). PENDING 캠프도 캠프 목록에 정상 노출·재진입되므로 고아 레코드 문제는 해소됐다. scenarios.md Feature 2-i로 시나리오 추가 완료. 다만 **코너·트랙이 하나도 없는 채로 "코너학습 시작"을 확정하면 어떻게 되는지**(경고만 하고 허용 vs 하드 블록)는 아직 미정이다 — 지금 목업은 막지 않는 쪽으로 가정했다.
- `POST /corners`(코너 일괄 생성)의 부분 실패 시 트랜잭션 범위(전체 롤백 vs 부분 반영)는 api-endpoints.md에 미정으로 남겨뒀다 — 결정되면 A0-b의 "상태" 항목에도 실패 UI(예: 실패한 행만 표시하고 재시도)를 추가해야 한다.
- A12 리포트의 시계열/운영지표 탭(analytics-model.md §1.5, §1.6)은 이번 세분화 범위에서 제외했다 — 필요성이 확인되면 탭을 2개 더 추가.
- A5/A13/A2B에 적용한 필터·정렬은 클라이언트 사이드 정렬로 가정했다(§design-system 4.5-b) — 조/트랙/로그 규모가 지금 가정(조 20, 트랙 10~20, 로그 수백 건)을 크게 넘어서면 서버 사이드 페이지네이션·정렬로 전환이 필요할 수 있다.
- B8/A3 트랙 교체 시 진행자 기기의 정확한 재인증 API 흐름은 api-endpoints.md에 "TBD"로 표기되어 있다(진행자 쪽 상세는 [screen-spec-facilitator.md](screen-spec-facilitator.md) B8 참고) — 확정 시 이 문서 A3 항목과 함께 갱신할 것.
