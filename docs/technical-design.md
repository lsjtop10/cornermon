# 기술 설계

> §domain-model.md / §scenarios.md / §analytics-model.md의 도메인 모델을 실제로 구현하기 위한 기술 스택과 아키텍처를 정의한다.

## 0-a. 백엔드 언어 결정 근거 (Go, 확정)

Kotlin(Spring Boot + JPA 리치 도메인 모델 + GraalVM AOT)과 비교 검토했다. Kotlin 쪽은 "복잡한 비즈니스 규칙(등록, 조배정 등 향후 확장)을 다루기 좋다"는 장점이 있고, AOT 네이티브 이미지로 JVM의 기동/메모리 단점도 상당 부분 상쇄 가능하다는 점까지 확인했다.

**그럼에도 Go로 최종 확정**했다. 근거:
- **마감이 타이트하다** — 지금 시점에 새 프레임워크(Spring Native/GraalVM AOT 빌드 파이프라인)를 처음 구축하는 학습·디버깅 비용을 감당하기 어렵다.
- **이미 익숙한 기술**이라 실제 개발 속도가 더 빠르다.
- Go 자체의 장점(명시적이고 간결한 코드, 성능, goroutine 기반 동시성 처리 용이성)이 이 프로젝트 규모에선 Java의 "비즈니스 규칙 표현력" 이점보다 크다고 판단.

**재론하지 않을 것**: 이후 세션에서 "Kotlin/Spring이 더 낫지 않냐"는 논의가 다시 나오면, 이 항목을 먼저 참조할 것 — 이미 트레이드오프를 다 검토하고 마감 압박을 이유로 의도적으로 Go를 선택한 결정이다.

## 0-b. DB 결정 근거 (PostgreSQL, 확정)

SQLite와 비교 검토했다. 이 프로젝트 규모(쓰기 200건 남짓, 몇 시간 분산)에서는 SQLite도 기능적으로 충분하다는 결론까지 갔다 — SQL 표준 기능(JOIN/CTE/윈도우함수/FK/부분 유니크 인덱스)은 거의 다 지원하고, 배포도 더 단순(단일 파일)하다.

**그럼에도 PostgreSQL로 최종 확정**했다. 근거:
- **PostgreSQL을 많이 운영해본 경험이 있다** — 개발 단계에서 성숙한 생태계(pgx, 도구, 레퍼런스)의 이점이 SQLite의 "배포 단순함" 이점보다 크다고 판단.
- `STDDEV` 등 통계 함수 내장, SQLite의 WAL/NFS 회피 같은 운영상 잔주의사항이 없음.
- Docker로 병행 구동하는 비용이 마감에 크게 부담될 정도는 아님 — SQLite 쪽이 "명백히 유리"한 상황은 아니었고 꽤 팽팽했던 트레이드오프.
- 향후 앱 인스턴스를 수평 확장할 가능성에도 미리 여유를 둠(SQLite는 이 시나리오에서 결국 마이그레이션이 필요).

**참고**: SQLite 채택 시 필요했을 것(참고용, 채택 안 함) — `VACUUM INTO` 기반 백업, WAL 모드+`synchronous=NORMAL`, 네트워크 마운트 파일시스템 회피, `STDDEV` 수동 구현(`SQRT(AVG(x*x)-AVG(x)*AVG(x))`).

## 0-c. 앱 프레임워크 결정 근거 (Flutter, 확정)

React Native와 비교 검토했다. 팀이 JavaScript/TypeScript와 Dart 둘 다 처음이라 "이미 익숙한 언어"라는, 여태까지 다른 결정(Go, PostgreSQL)의 결정적 근거가 이번엔 무승부라 순수 기술적 적합성으로 판단했다.

**Flutter로 확정**했다. 근거:
- **SSE가 이 앱의 핵심 요구사항(§2.3)인데, Flutter가 이 지점에서 실질적으로 유리하다.** React Native는 `EventSource`를 기본 지원하지 않아 서드파티 폴리필(`react-native-sse`)이 필요하고 백그라운드/포그라운드 전환 시 엣지케이스가 있다. Flutter는 Dart의 `Stream`이 SSE의 연속 이벤트 모델과 자연스럽게 맞아떨어져 구현이 더 직접적이다.
- **빌드/운영 안정성**: RN은 네이티브 브릿지 버전 불일치로 업그레이드가 까다로웠던 전례가 있다(New Architecture로 개선 중이나 완전히 해소되진 않음). Flutter는 자체 렌더링 엔진(Skia/Impeller)을 쓰기 때문에 이 클래스의 문제가 원천적으로 적고, 빌드가 더 결정적(deterministic)이다.
- **Dart는 정적 타입 언어**라, Go에 익숙한 지금 배경과 결이 비슷해 학습 곡선이 JS/TS 생태계(동적 타이핑, 패키지 파편화)보다 상대적으로 완만할 가능성이 있다.
- **트레이드오프로 잃는 것**: RN+Expo가 제공하는 OTA 업데이트(JS 변경분을 스토어 재심사 없이 배포)만큼 성숙한 대안이 Flutter엔 없다(Shorebird 등 서드파티는 상대적으로 덜 성숙). 행사 직전 급한 수정 배포가 필요하면 이 부분이 아쉬울 수 있다 — 다만 이 앱은 소수 기기에 비공개 배포(TestFlight/내부 트랙)라 스토어 심사 지연 자체가 크지 않아 감내 가능하다고 판단.

## 0-d. 관리자 앱 / 진행자 앱 코드베이스 구조 (단일 레포, 멀티 엔트리포인트 — 확정)

두 클라이언트는 화면 구성이 완전히 다르지만(관리자=iPad 대시보드, 진행자=모바일 스캔 앱, §screen-spec.md), 도메인 모델·API 클라이언트·인증 토큰 처리·디자인 토큰(§design-system.md)은 100% 동일하다. 이 중복을 피하면서도 두 앱을 별개의 배포 산출물로 유지하기 위해 **Flutter 프로젝트 1개 + 멀티 엔트리포인트** 구조를 쓴다.

- **패키지 구조**:
  ```
  /lib
    main_admin.dart          # 관리자 앱 진입점 — runApp(AdminApp())
    main_facilitator.dart    # 진행자 앱 진입점 — runApp(FacilitatorApp())
    /common
      /domain                # 서버 domain 모델에 대응하는 클라이언트 측 모델(Group, Corner, Track, Visit ...)
      /api                   # REST 클라이언트, SSE 구독(EventSource 격 Stream 래퍼)
      /auth                  # 불투명 토큰 저장(Keychain/Keystore), 토큰 갱신 로직
      /design_system         # 컬러/타이포/스페이싱 토큰(design-system.md 1:1 매핑), 공유 위젯(버튼/뱃지/모달)
    /admin
      /screens                # A0~A14 (screen-spec.md Part A)
      /widgets                # 관리자 전용 위젯(상태 카드 그리드, 사이드바 등)
    /facilitator
      /screens                # B0~B8 (screen-spec.md Part B)
      /widgets                # 진행자 전용 위젯(PIN OTP 입력, QR 스캔 프레임 등)
  ```
- **빌드 분리**: `flutter run -t lib/main_admin.dart --flavor admin` / `flutter run -t lib/main_facilitator.dart --flavor facilitator` 로 완전히 독립된 두 바이너리를 뽑는다. iOS는 Xcode scheme, Android는 `build.gradle` productFlavors로 앱 아이콘·번들 ID·표시 이름을 분리한다 — 최종 사용자(App Store/TestFlight 기준)는 "코너학습 관리자", "코너학습 진행자" 두 개의 별개 앱으로 인지한다.
- **`/common`을 가로지르는 규칙**: `/admin`, `/facilitator` 디렉터리는 서로를 import하지 않는다(단방향 의존 금지 — 두 앱이 서로의 화면/위젯을 참조하기 시작하면 결국 하나의 거대 앱이 되어 이 구조를 쓰는 의미가 없어진다). 공유가 필요한 것은 반드시 `/common`으로 승격시킨 뒤 양쪽에서 참조한다.
- **왜 완전 별도 레포(멀티 프로젝트)가 아닌가**: 이 규모(개발 인원 소수, 도메인 모델이 자주 바뀌는 초기 단계)에서 별도 레포 2개 + 공유 pub 패키지(예: melos 모노레포)는 초기 셋업 비용과 "공유 패키지 버전을 양쪽에서 맞춰야 하는" 동기화 비용이 더 크다고 판단했다. 단일 레포면 도메인 모델 변경이 한 커밋으로 양쪽에 동시 반영되고 컴파일 타임에 불일치가 바로 드러난다.
- **왜 완전 단일 앱(런타임 역할 분기)이 아닌가**: 로그인 역할에 따라 화면 트리만 바꾸는 방식은 배포 산출물이 1개라는 장점은 있으나, 진행자 기기에 관리자 기능 코드가 그대로 번들링된다 — 관리자 전용 기능(감사 로그, 규칙 변경, 기기 승인 등)이 진행자 기기의 앱 바이너리 안에 존재하는 것 자체가 불필요한 공격 표면이며, §domain-model.md 4.4의 "역할 기반 접근 제어"(서버 API 레벨 스코프 제한)와 별개로 클라이언트 바이너리 레벨에서도 분리하는 편이 원칙에 맞는다고 판단해 채택하지 않았다.

## 1. 아키텍처 원칙

### 1.1 두 가지만 지킨다
1. **도메인 순수성**: `domain` 패키지의 타입(Group, Corner, Track, Visit, Message 등)은 어떤 프레임워크/DB/HTTP 타입도 import하지 않는다. 순수 Go struct + 비즈니스 로직 메서드만 존재한다.
2. **의존성 역전**: 도메인/유스케이스 계층은 "무엇이 필요한지"를 인터페이스로 선언하고, 그 구현은 바깥(adapter) 계층이 제공해 주입한다. 도메인이 DB나 HTTP를 직접 알지 못한다.

이 두 원칙만 지키면 되고, 엔티티/유스케이스/인터페이스어댑터/프레임워크의 4계층을 형식적으로 다 나누지 않는다.

### 1.2 패키지 구조 (제안)
```
/internal
  /domain          # 순수 도메인: Group, Corner, Track, Visit, Message, Camp ... + 비즈니스 규칙 메서드
  /usecase         # 유스케이스(애플리케이션 서비스): "시작 스캔 처리", "트랙 교체 처리" 등
                   #   - 여기서 필요한 포트(인터페이스)를 정의한다: VisitRepository, TrackRepository, Broadcaster ...
  /adapter
    /postgres      # usecase가 정의한 Repository 인터페이스의 PostgreSQL 구현체
    /http          # REST 핸들러(진행자 앱, 관리자 앱, 관리자 웹 등 클라이언트 진입점)
    /sse           # 실시간 push 구현체 (Broadcaster 인터페이스 구현)
    /push          # 필요 시 모바일 푸시(FCM/APNs) 구현체
  /config, /main.go
```

- **쿼리 튜닝이 필요한 지점**: 범용 Repository 인터페이스로 퉁치지 않고, 그 유스케이스 전용의 좁은 인터페이스를 만든다. 예: 캠프 종료 시점의 "캠프 결과 리포트"(§analytics-model.md)처럼 무거운 집계 쿼리가 필요하면 `usecase` 계층에 `ReportQuerier` 같은 좁은 인터페이스를 선언하고, `adapter/postgres`에 손튜닝된 SQL로 구현해 주입한다. 일반 CRUD성 Repository와 이런 튜닝된 Querier를 같은 인터페이스에 억지로 합치지 않는다.

## 2. 도메인 → 구현 매핑

### 2.1 핵심 엔티티 (Go struct, `domain` 패키지)
`Camp`, `Group`, `Corner`, `Track`, `Visit`, `Facilitator Session`(트랙 세션), `Admin`, `DeviceRegistration`(PENDING/APPROVED/REJECTED/REVOKED 토큰), `Message`(Broadcast/Direct), `AuditLog`.

각 엔티티는 §domain-model.md에 정의된 불변식을 **자기 자신의 메서드**로 강제한다. 예:
```go
// domain/track.go (개념 예시, 실제 코드 아님)
func (t *Track) StartVisit(group GroupID, now time.Time, method InputMethod) (*Visit, error) {
    if t.Status != TrackActive {
        return nil, ErrTrackNotActive
    }
    if t.CurrentVisit != nil {
        return nil, ErrTrackBusy // §2.2 동시 진행 제약
    }
    ...
}
```
DB 트랜잭션 경계는 `usecase` 계층이 잡고, 위반 시 롤백한다 — 도메인 메서드는 저장을 모른다.

### 2.2 인증/기기 신뢰
- 트랙 PIN, 기기 등록 토큰(§domain-model.md 2.4-b) 모두 **무작위 값 생성 후 DB에는 해시만 저장**(비밀번호 저장과 동일 원칙 — 평문 토큰은 클라이언트에만 존재).
- 기기 토큰 상태(PENDING/APPROVED/REJECTED/REVOKED)는 도메인 상태이므로 `domain.DeviceRegistration`에 상태 전이 메서드로 구현하고, `usecase`가 저장을 오케스트레이션한다.

### 2.2-a JWT가 아니라 불투명 토큰(opaque token) — 전 인증 계층 공통 원칙 (확정)
트랙 PIN 세션, 기기 신뢰 토큰, 관리자 액세스/리프레시 토큰(§2.2-b) **전부 JWT가 아니라 불투명 토큰**으로 통일한다. 클라이언트는 `Authorization: Bearer <무작위 문자열>`로 제시하고, 서버는 매 요청마다 그 토큰(의 해시)을 DB에서 조회해 상태·소속·만료를 확인한다.

- **네이티브 앱이라 "쿠키냐 JWT냐"는 애초에 잘못된 이분법**이다 — 쿠키는 브라우저 전송 메커니즘(자동 첨부, SameSite/CSRF)이라 네이티브 앱엔 해당 없음. 실제 축은 "자기서명 JWT(무상태)" vs "불투명 토큰(서버 조회 필요)"이다.
- **이 도메인은 즉시 무효화 요구가 도처에 있다**: 트랙 삭제→세션 종료, 기기 신뢰 회수→PIN 화면 차단, 관리자 강제 로그아웃/세션 회수, PIN 재생성→구 PIN 무효화. JWT는 서명 검증만으로 유효성을 판단하는 게 핵심 장점인데, 이를 살리면 발급된 토큰을 만료 전에 되돌릴 방법이 없다(무효화 목록을 어차피 DB에 두고 매번 대조해야 한다면 JWT의 장점 자체가 사라짐).
- **이 규모(동시 접속 20개 안팎)에서 매 요청 DB 조회는 완전히 무시할 비용**이라, JWT가 피하려는 "DB 조회 비용 절감"이라는 이점 자체가 우리에겐 손해가 아니다. 반대로 불투명 토큰은 상태 필드 하나만 바꾸면 다음 요청부터 즉시 반영되므로, 위 무효화 요구사항들과 정확히 맞아떨어진다.
- **JWT가 원래 풀려던 문제가 아니다**: JWT의 무상태 검증은 "서로 DB를 공유하지 않는 여러 서비스가 각자 독립적으로 검증해야 할 때"(마이크로서비스 간 검증, OAuth2/OIDC 같은 조직 경계를 넘는 신원 위임)처럼 검증 주체가 여러 곳으로 흩어질 때 발휘된다. 우리는 Go 단일 프로세스·단일 서비스라 이 문제 자체가 없다. 게다가 탈취 대비를 위해 액세스 토큰을 짧게 두고 리프레시 토큰으로 갱신하는 흔한 패턴을 쓰면, 결국 리프레시 토큰은 서버가 상태로 추적/회수해야 하므로 "무상태"라는 JWT의 핵심 이점이 다시 사라진다 — 즉 대부분의 실전 인증 요구(서버 측 즉시 통제)와 JWT의 무상태성은 원래 상충하는 목표다.

### 2.2-b 관리자 세션: 액세스/리프레시 토큰 이원화 (확정)
관리자는 개인 계정(ID/비밀번호)이라 탈취 시 파급력이 진행자보다 크므로(전체 조회+규칙변경+PIN조회 등), 진행자처럼 "자동 만료 없음"을 그대로 적용하지 않는다. 그렇다고 짧은 만료 = 잦은 재로그인 요구는 아니다 — **재로그인(ID/비밀번호 재입력)과 세션 재획득(토큰 재발급)을 분리**한다.

- **액세스 토큰**: (§2.2-a에 따라 불투명 토큰) 수명 15~30분. 실제 API 호출마다 제시하는 토큰. 탈취 시 노출 창을 짧게 유지하는 게 목적이며, 필요하면 즉시 회수도 가능하다(JWT였다면 자연 만료까지 기다려야 함).
- **리프레시 토큰**: 슬라이딩 만료(사용할 때마다 시계 리셋, 예: 최종 사용 후 12시간 미사용 시 만료). 기기 보안 저장소(Keychain/Keystore)에만 저장하고, 오직 "액세스 토큰 재발급" 전용 엔드포인트에만 제시한다 — 일반 API 호출에는 쓰지 않는다. 진행자 기기 신뢰 토큰(§domain-model.md 2.4-b)과 동일한 발급/저장/해시 원칙을 그대로 재사용한다.
- **자동 갱신(silent refresh)**: 액세스 토큰 만료 직전 또는 401 응답 시, 앱이 사용자 개입 없이 리프레시 토큰으로 새 액세스 토큰을 받아온다. **관리자가 앱을 계속 쓰는 한 로그인 화면을 다시 볼 일이 없다.**
- **실제 재로그인이 필요한 경우**: 리프레시 토큰 자체가 만료됐거나(장시간 미사용), 관리자가 그 기기의 세션을 명시적으로 회수(revoke)한 경우뿐이다.
- **세션 회수**: 관리자 2명은 완전 동등 권한(§domain-model.md 2.5-b)이므로, 서로의 리프레시 토큰도 "세션 관리" 화면에서 회수할 수 있다 — 기기 분실 대응.

### 2.3 실시간 갱신 (SSE)
- 관리자 대시보드, 진행자 앱(공지/다이렉트 메시지 수신)은 `GET /events`류 SSE 연결을 유지한다.
- 서버 내부에서 상태 변화(방문 시작/종료, 트랙 생성/삭제/교체, 메시지 발송)가 생기면 `usecase`가 도메인 이벤트를 만들고, `adapter/sse`의 `Broadcaster` 인터페이스 구현체가 연결된 클라이언트에게 push한다.
- 클라이언트 액션(시작 스캔, 메시지 전송, 규칙 변경 등)은 평범한 `POST /api/...`로 처리한다 — 이 흐름에는 실시간성이 필요 없다(요청-응답으로 충분).
- **2초 이내 반영**: SSE는 서버가 변화를 감지하는 즉시 push하므로 사실상 수백 ms 내에 반영된다. 폴링이 아니므로 "2초"라는 여유 요구사항을 가뿐히 만족하며, 오히려 폴링(1초 간격 등)보다 서버 부하가 낮다(트랙 12~14개 규모에서는 의미 있는 차이는 아니지만).
- 네트워크 단절 시 SSE는 표준적으로 자동 재연결을 지원하므로(`EventSource` 스펙), 클라이언트 구현 부담이 적다. React Native에서는 SSE 폴리필이 필요할 수 있어 이 부분은 프론트 기술검토 시 확인 필요.

### 2.3-b 실시간 push와 DB 상태의 정합성 보장
쓰기 부하 자체는 무시할 수준(캠프 전체 200건 남짓, 몇 시간에 분산)이지만, "DB에 저장된 값"과 "관리자가 실시간으로 보는 값"이 어긋나지 않도록 아래 규칙을 지킨다.

1. **브로드캐스트는 커밋 성공 직후에만, 순차적으로**: `usecase`는 `DB 트랜잭션 커밋 → (성공 시에만) Broadcaster 호출`을 반드시 같은 흐름에서 순서대로 실행한다. 커밋 전에 낙관적으로 브로드캐스트하지 않는다 — 롤백된 변경이 관리자 화면에는 반영되는 "유령 이벤트"를 방지하기 위함.
2. **이벤트는 델타가 아니라 그 시점의 전체 스냅샷을 담는다**: 예를 들어 "트랙 상태 변경" 이벤트는 "BUSY로 바뀜" 같은 델타 대신 그 트랙(또는 코너)의 현재 상태 전체를 담아 보낸다. 이 규모(트랙 14개)에서는 페이로드 비용이 무시할 수준이고, 이벤트를 하나 놓쳐도 다음 이벤트로 자동 정합된다(델타 누락 시 영구히 어긋나는 문제를 원천 차단).
3. **(재)연결 시 첫 메시지로 항상 현재 전체 스냅샷을 push한다**: 별도의 "REST로 초기 조회 → SSE 구독"이라는 2단계를 두지 않는다. SSE 연결이 맺어지는 순간(최초 연결이든, 네트워크 끊김 후 자동 재연결이든 동일하게) 서버는 그 즉시 현재 전체 상태를 첫 메시지로 push하고, 이후 변경이 생길 때마다 같은 방식으로 스냅샷을 push한다. 이러면 "초기조회와 구독 사이의 틈새"라는 문제 상황 자체가 성립하지 않는다 — 하나의 연결 안에서는 전송 순서가 보장되므로(TCP), 별도의 시퀀스 번호나 gap-감지 로직이 필요 없다. 끊기면 재연결이 곧 리싱크다.
4. **좀비 연결(zombie connection) 탐지**: 클라이언트는 연결됐다고 착각하지만 실제로는 죽은 연결(중간 프록시/NAT가 조용히 끊었는데 FIN/RST가 전달 안 된 경우 등) — 이 상태에서는 `EventSource`의 자동 에러/재연결이 아예 안 뜰 수 있다. 대응:
   - **서버**: 각 SSE 연결에 15~20초 주기로 SSE 하트비트(`:` 로 시작하는 주석 라인, 이벤트로 처리되지 않음)를 write한다. 죽은 연결에 쓰면 즉시 실패하므로 서버가 그 즉시 구독자 목록에서 정리할 수 있다.
   - **클라이언트**: 마지막 수신 시각(하트비트 포함)을 기록해두고, 하트비트 주기의 2배 정도(예: 40초) 침묵이 지속되면 브라우저/앱의 자동 재연결을 기다리지 않고 **스스로 연결을 끊고 새로 연다**.
   - **책임 소재**: 서버는 누가 무엇을 받았는지 추적하지 않는 무상태(stateless) 브로드캐스트만 한다. "내가 침묵 상태인가"를 감지하고 재연결(=재동기화, 특정 메시지의 재전송이 아니라 최신 스냅샷을 통째로 새로 받는 것)하는 책임은 전적으로 클라이언트에 있다.
   - **최후 안전망**: 위 메커니즘도 놓치는 극단적 상황을 위해 관리자 대시보드가 30초 주기로 REST 전체 재조회를 추가로 수행한다. 코너/트랙 수가 10~20개 수준이라 이 정도 재조회 비용은 무시할 만하다.
5. **트랜잭셔널 아웃박스 패턴은 필요 없음**: 정확히는 "전체 스냅샷 방식이라서"가 아니라 — 스냅샷 방식 덕분에 **개별 메시지의 전달 보장 자체가 불필요**해졌기 때문이다(메시지 하나를 놓쳐도 다음 스냅샷이나 재연결/재조회가 자동으로 따라잡음). 델타 방식이었다면 메시지 하나 유실이 곧 영구적 불일치로 이어지므로 아웃박스 같은 전달 보장 장치가 필요했을 것이다.

### 2.4 데이터베이스 스키마 방향 (요약, 상세 DDL은 별도)
- `camps`, `groups`, `corners`, `tracks`(corner_id FK, 상태), `visits`(group_id, track_id FK, started_at, ended_at, input_method), `device_registrations`(token_hash, status), `messages`(channel_type, track_id nullable, sender), `audit_logs`.
- 주요 유니크 제약: `visits`에 `(group_id, corner_id) WHERE status = 'COMPLETED'`에 대한 부분 유니크 인덱스로 "중복 방문 금지"를 DB 레벨에서도 강제(애플리케이션 로직과 이중 방어).
- **토큰 조회 성능**: §2.2-a의 불투명 토큰(`token_hash`) 컬럼에는 유니크 인덱스를 건다 — 이 규모에서는 사실상 O(1) 조회라 병목이 되지 않는다. 게다가 SSE 연결은 **연결 시점에 1회만 인증**하고 연결이 유지되는 동안 재인증하지 않으므로(REST POST 액션에만 매 요청 토큰 조회가 발생), 인증 관련 DB 조회 총량 자체가 적다.
- **향후 확장 시**: 트래픽이 커지면 토큰 조회 앞단에 Redis 같은 인메모리 KV 캐시를 얹는 선택지가 있다. `usecase`가 Repository를 인터페이스로만 의존하므로(§1.2), PostgreSQL 직접 조회 구현체를 캐시-우선 구현체로 교체하는 정도로 대응 가능하다 — 지금 규모에서는 불필요.
- `tracks.corner_id`는 생성 후 업데이트하지 않는다(§domain-model.md 2.2) — 애플리케이션 레벨 규칙이지만, 원한다면 트리거로도 강제 가능(과설계일 수 있어 우선 애플리케이션 레벨만).
