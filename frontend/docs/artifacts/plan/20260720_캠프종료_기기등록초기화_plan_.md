### 종료 조건

- `camp_ended`는 종료를 빠르게 반영해 기기 등록과 PIN 세션을 정리하는 terminal 신호다. 단, 이벤트의 순서·수신 여부는 종료 원인의 정합성 근거가 아니다.
- 일반 이벤트의 REST 재조회가 `SESSION_REVOKED`/401로 실패하면 `GET /device-registrations/me`를 호출하고, 응답의 `status`와 `campStatus`로 최종 처리한다.
- `APPROVED + ACTIVE`이면 PIN 세션만 제거해 PIN 로그인 화면으로 이동한다. `REVOKED + ENDED` 또는 `REVOKED + ACTIVE` 등이면 기기 신뢰 토큰·등록 ID·로컬 상태와 저장된 트랙 세션을 제거해 등록 화면으로 이동한다.
- SSE 유실 또는 일반 이벤트 재조회 401에서의 영속 상태 정합성은 위 REST 결과로만 확정한다.
- 백엔드가 `campStatus` 계약을 구현·전달한 뒤 OpenAPI 생성 코드를 갱신한다.

## 2. 설계

신뢰 해제의 단일 책임은 `DeviceTrust` notifier에 둔다. 기존의 REVOKED 폴링 처리, `camp_ended` 빠른 처리, **기기 상태 조회의 REVOKED 판정**, PIN 실패, 사용자 액션이 모두 이 메서드를 호출해 저장소 키와 provider 상태가 어긋나지 않게 한다.

```dart
@riverpod
class DeviceTrust extends _$DeviceTrust {
  /// 등록 코드 입력을 다시 받을 수 있도록 로컬 신뢰 정보를 원자적으로 초기화한다.
  Future<void> clearRegistration();
}

Future<void> _recoverFromSessionRevoked() async {
  final registration = await deviceRegistrationApi.getMyRegistration();
  if (registration.status == DeviceStatus.revoked) {
    await ref.read(deviceTrustProvider.notifier).clearRegistration();
  } else {
    ref.read(trackSessionProvider.notifier)
        .handleTermination(TrackSessionTerminationReason.sessionRevoked);
  }
}
```

- `clearRegistration()`은 폴링을 중지하고 `device_trust_token`, `device_trust_status`, `device_trust_registration_id`를 지우며 `deviceTrustTokenProvider`를 invalidate한다.
- 상태를 `none`으로 전이하면 기존 `facilitator_router.dart`의 최우선 가드가 모든 진행자 화면에서 `/device-pending`으로 보낸다.
- 수동 초기화는 서버 기기 등록을 임의로 REVOKED로 바꾸지 않는다. 서버의 신뢰 철회는 캠프 종료/관리자 회수의 권한으로 유지하고, 이 UI는 해당 기기의 **로컬 등록 정보만** 지운다.
- `camp_ended`는 terminal fast path로 `clearRegistration()`과 세션 종료를 즉시 수행한다. 반면 `camp_updated` 등 일반 이벤트의 기존 REST 재조회와 이후 401 복구는 SSE를 놓쳤을 때 같은 최종 상태에 수렴시키는 권위 있는 경로다.

### 권한 상실 복구 표

| 계기 | 기기 상태 | 캠프 상태 | 처리 |
| --- | --- | --- | --- |
| 일반 SSE 후 REST 재조회 실패, 또는 이후 보호 REST 401 | `APPROVED` | `ACTIVE` | PIN 세션만 제거하고 PIN 로그인 화면으로 이동한다. |
| 동일 | `REVOKED` | `ENDED` | 캠프 종료로 판정한다. 기기 등록과 PIN 세션을 모두 제거하고 등록 화면으로 이동한다. |
| 동일 | `REVOKED` | `ACTIVE` 등 | 기기 신뢰 회수로 판정한다. 기기 등록과 PIN 세션을 모두 제거하고 등록 화면으로 이동한다. |

## 4. API 계약 영향

백엔드가 기존 `GET /device-registrations/me` 응답에 아래 필드를 추가한 계약을 먼저 구현·전달한다. 프론트는 전달받은 OpenAPI 계약으로 생성 코드를 갱신한 뒤 소비한다.

```json
{
  "id": "device-1",
  "campId": "camp-1",
  "status": "REVOKED",
  "campStatus": "ENDED"
}
```

`campStatus`는 `PENDING | ACTIVE | ENDED`이고, 이 변경은 API Change Protocol에 따라 백엔드 계약 PR을 기준으로 한다.

## 5. 검증 체크리스트

- [x] `camp_updated`는 현재 방문 REST 재조회를 유발하고, 보호 REST 401과 PIN `DEVICE_NOT_TRUSTED`는 기기 상태 조회로 최종 판정한다. `camp_ended`는 terminal fast path로 즉시 정리한다.
- [x] `APPROVED + ACTIVE`는 기기 등록을 보존하고, `REVOKED + ENDED` 및 REVOKED의 그 외 캠프 상태는 등록을 삭제한다.
- [x] 수동 등록 취소는 확인 후에만 로컬 키를 삭제하고, 취소하면 현재 화면 상태를 유지한다.
- [x] PENDING 승인 폴링과 관리자 REVOKED 폴링의 기존 동작이 유지된다.
- [x] 백엔드가 전달한 OpenAPI 계약으로 `lib/shared/api/gen`을 갱신하고 formatter·분석·관련 테스트를 실행했다.
