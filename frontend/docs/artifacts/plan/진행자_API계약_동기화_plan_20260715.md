# 진행자(facilitator) API 계약 동기화 Plan

## 0. 배경 / 문제 정의

`cornermon_api_gen` 재생성 이후 진행자(facilitator) 쪽 코드가 옛 계약을 가정한 채 남아 있어
`dart analyze lib/facilitator test/facilitator` 기준 **160개 에러**가 발생한다. 관리자(admin)
테스트 스위트는 정상이지만, 같은 `flutter test`/`flutter analyze` 실행 시 진행자 컴파일 에러가
전체를 막아 관리자 작업에도 지장을 준다.

이미 일부 커밋되지 않은 수정이 진행 중이었다(`git status`):
`entities/group_ext.dart`, `entities/track_ext.dart`, `session/device_trust_provider.dart`,
`session/track_session_provider.dart`, `shared/api/domain_aliases.dart`. 이 Plan은 그 작업을
이어받아 **남은 전체 범위**를 다룬다.

### 🆕 발견 → 해결 — 기기 신뢰 토큰 부착 경로

처음엔 `deviceToken`이 바디로 오게 됐어도 이걸 저장/부착하는 코드가 어디에도 없다는 문제를
발견했었다(§구 버전 기록: `AuthInterceptor`가 단일 `SessionTokenSource.currentAccessToken`만
보고 `Authorization` 헤더를 다는데, 트랙 세션은 로그인 전엔 토큰이 없어 PIN 로그인 자체가
403이 나는 구조였음).

**그런데 `0e8feba` 커밋으로 `TrustedDeviceAuth`가 아예 별도 헤더(`X-Device-Token`)로
분리되면서 문제가 훨씬 단순해졌다.** 재생성된 코드를 확인한 결과:

```dart
Future<Response<TrackLoginResponse>> authTrackLoginPost({
  required String xDeviceToken,   // ← 신규, named parameter로 직접 노출
  required TrackLoginRequest request,
  ...
})
```

즉 **공용 `AuthInterceptor`/`SessionTokenSource`는 건드릴 필요가 없다** — 이 토큰이 필요한
호출은 `/auth/track/login` 단 하나뿐이고, 생성된 클라이언트가 이를 일반 파라미터로 노출하므로
`track_session_provider.dart`의 `loginWithPin()`에서 저장해둔 값을 직접 넘기면 끝난다.

**필요한 수정** (Phase A):
1. `device_trust_provider.dart`의 `requestRegistration()`에서 `response.data.deviceToken`을
   secure storage에 저장(신규 키, 예: `_deviceTrustTokenKey`). 같은 파일에 저장된 값을 읽어오는
   `@riverpod Future<String?> deviceTrustToken(Ref ref)` provider도 함께 추가(기존
   `facilitatorCampId`와 동일한 패턴).
2. `track_session_provider.dart`의 `loginWithPin()`을 다음과 같이 수정:
   ```dart
   Future<void> loginWithPin(String pin) async {
     final deviceToken = await ref.read(deviceTrustTokenProvider.future);
     if (deviceToken == null) throw Exception('신뢰된 기기가 아닙니다.');
     final api = ref.read(authDeviceTrustApiProvider);
     final response = await api.authTrackLoginPost(
       xDeviceToken: deviceToken,
       request: AuthTrackLoginPostRequest((b) => b..pin = pin),
     );
     ...
   }
   ```

### 백엔드 후속 작업(프론트 범위 밖, 참고용)

- `GET /camps/{campId}/messages/broadcast`에 `TrackAuth` 허용 필요(B6 구현의 전제조건).
  스웨거 갱신 + `docs.go`/`swagger.json` 재생성 후 프론트에서 다시 codegen 재생성.

### 계약이 바뀐 지점 (조사 결과)

1. **거의 모든 응답 모델 필드가 nullable로 바뀜** (`String? id`, `int? trackNo`,
   `DateTime? startedAt` 등). 옛 코드는 non-null을 가정하고 작성됨.
2. **POST 요청 바디 파라미터명이 `request`로 통일**됨
   (예전 `deviceRegistrationsPostRequest:` → `request:`, `authTrackLoginPostRequest:` → `request:`).
3. **`groupList`, `broadcastMessageList` family provider가 `CampId`를 요구**하게 됨
   (`shared/api/providers/group_providers.dart:17`, `message_providers.dart:16`).
   단, 두 provider의 사정이 다르다는 걸 재확인했다:
   - **조 목록(B4 수동체크인)은 campId가 애초에 필요 없다.** `swagger.yaml`에
     `GET /tracks/{trackId}/groups`("인증된 진행자의 트랙이 속한 캠프의 조 목록", `TrackAuth`)가
     이미 존재하고, `group_providers.dart`에 `trackScopedGroups(Ref, TrackId)`로 이미 구현되어
     있다. `manual_checkin_screen.dart`가 이 provider 대신 camp 전체용
     `groupListProvider(CampId)`를 잘못 호출하고 있었을 뿐이다 — **campId 브리징 자체가 불필요.**
   - **공지 목록(B6)은 진짜 문제다.** `GET /camps/{campId}/messages/broadcast`가
     `security: AdminAuth`만 허용해서, campId를 안다 해도 진행자 세션으로는 애초에 호출이
     막혀 있었다. 위 "전제조건"에 정리한 백엔드 변경(TrackAuth 허용 + `CornerResponse.campId`
     추가)이 이 문제의 실제 해법이고, 로컬에 campId를 캐싱해서 브리징하는 방식은 필요 없다.
4. **SSE `SSENotification.scope`가 문자열(`"track:xxx"`/`"broadcast"`)에서 구조화 객체
   `SSEScope{kind: camp|track, trackId}`로 바뀜**. 또한 `event.data?.scope`가 아니라
   `event.scope`로 최상위 필드다(`data` 필드 자체가 없어짐).
5. **`TrackDirectActions.send()`가 존재하지도 않는 `MessagesBroadcastPostRequest`를
   `tracksTrackIdMessagesPost`에 넘기고 있음** — 실제로는 `DirectMessageRequest`가 필요하고,
   이미 `shared/api/providers/message_providers.dart`에 올바른 `sendDirectMessage` family
   provider가 구현되어 있다(중복 구현 금지 원칙 — `workflow/plan.md` §3.2).

이 중 1~2번은 단순 치환/null 처리로 끝나지만, 3~5번은 **로직 흐름을 새로 설계**해야 한다
(사용자가 지적한 지점과 일치).

### 2.2 (신규) `facilitator/session/facilitator_broadcast_provider.dart`

**책임**: (백엔드가 `GET /camps/{campId}/messages/broadcast`에 `TrackAuth`를 허용한 뒤)
`broadcastMessageListProvider(CampId)`를 진행자 화면이 지금까지 써온 "인자 없는" 형태로
다리 놓는다. campId의 출처는 로컬 캐시가 아니라 **현재 인증된 트랙 세션의 `corner.campId`**
— 별도 영속화 없이
세션과 항상 동기화되고, 기기가 재등록되어도 stale해질 여지가 없다.

```dart
// 책임: 인증된 세션에서 campId를 얻어 broadcastMessageListProvider(CampId)에 위임.
// 미인증/campId 없음은 라우터 가드가 정상 동작하면 도달하지 않는 상태 — 방어적으로 빈 목록 반환.
@riverpod
Future<List<Message>> facilitatorBroadcastMessageList(Ref ref) async
```

`broadcastMessageListProvider`(인자 없이 잘못 참조되던 것)를 쓰던 3개 파일
(`broadcast_inbox_screen.dart`, `_main_track_header.dart`, `track_event_coordinator.dart`)은
이 provider로 교체한다. `ref.invalidate(...)` 호출부도 동일하게 바꾼다.

`manual_checkin_screen.dart`의 조 목록은 위 §0 조사대로 campId가 필요 없으므로, 별도 provider
없이 기존 `trackScopedGroupsProvider(trackId)`(`group_providers.dart`에 이미 구현됨)로 직접
교체한다.

> `device_trust_provider.dart`에 이미 추가돼 있는 `facilitatorCampIdProvider`(기기등록 시
> 로컬 저장소에 campId를 캐싱)는 이 문제 해결에는 더 이상 필요 없다. 다른 화면(예: 기기승인
> 대기 화면에 캠프명 표시 등)에서 이미 쓰이고 있지 않다면 제거를 권장하되, 새로 확인된
> 사용처가 없으므로 이 Plan에서 임의로 되돌리지 않고 Phase A에서 "제거 여부 확인" 항목으로만
> 남긴다.

### Phase C — SSE 분기 로직 재설계 (P1)

`track_event_coordinator.dart`의 `_handle`:

```dart
void _handle(TrackId trackId, SseEvent event) {
  final scope = event.scope;
  final isThisTrack = scope?.kind == SseScopeKind.track &&
      scope?.trackId == trackId.value;
  switch (event.event) {
    case SseEventEventEnum.trackUpdated:
      if (isThisTrack) ref.invalidate(currentVisitProvider(trackId));
    case SseEventEventEnum.messagesChanged:
      if (scope?.kind == SseScopeKind.camp) {
        ref.invalidate(facilitatorBroadcastMessageListProvider);
      } else if (isThisTrack) {
        ref.invalidate(trackMessageListProvider(trackId));
      }
    case SseEventEventEnum.trackDeleted: ...  // 변경 없음
    ...
  }
}
```

**확인 필요(사용자 확인 요망)**: 공지(broadcast) 메시지 갱신 이벤트의 scope.kind가
`camp`로 온다는 전제는 `api/openapi.yaml`의 `/events/track/{trackId}` 설명과
`SSEScope.kind` enum(`camp`, `track`) 값에서 추론한 것이지, 백엔드 실제 발행 코드까지
확인한 것은 아니다(프론트 스코프 작업 원칙상 백엔드 코드는 보지 않음 — CLAUDE.md
"scope of work"). 구현 단계에서 실기기/스테이징 통합 테스트로 반드시 재확인한다.
-> 추론이 맞아.

### Phase D — 테스트 스위트 (P1)

`test/facilitator/**` 10개 파일이 동일한 패턴의 에러를 공유한다:

1. **Fixture 생성자 문제**: `Message(...)`, `Group(...)`, `Track(...)`, `Corner(...)`,
   `VisitSummary(...)`, `SseEvent(...)` 등이 "undefined function"으로 잡힘 — production 코드와
   동일하게 `domain_aliases.dart` import 누락이 원인. 해결책은 동일(import 교체).
2. **Fake API 클래스 시그니처 불일치**: `pin_login_test.dart`의 `_FakeAuthDeviceTrustApi`가
   `authTrackLoginPostRequest:` named param을 오버라이드하는데 실제 인터페이스는 `request:`다
   → `invalid_override`. `broadcast_inbox_test.dart`의 `_FakeMessagesApi`는
   `EMessageApi`를 extend하는데(`extends_non_class`) 실제로는 `class`가 아니라 override 대상이
   바뀌었을 가능성 높음 — 제너레이터 산출물(`e_message_api.dart`)의 현재 시그니처 기준으로
   Fake를 다시 작성해야 한다.
3. **Provider override 문제**: `broadcastMessageListProvider`가 family가 되면서
   `overrideWith((ref) => fixture)` 형태가 `(Ref, CampId) => ...`를 요구하게 됨. Phase A/B에서
   `facilitatorBroadcastMessageListProvider`(무인자)로 화면 쪽을 바꾸면, 테스트도 이 무인자
   provider를 override하는 편이 화면 코드와 일치해 더 간단하다(백엔드 계약 변경 후 진행).
4. **not_enough_positional_arguments**: `groupListProvider()` 호출부(테스트 내 직접 호출) →
   `trackScopedGroupsProvider(trackId)`로 교체(campId override 불필요).
