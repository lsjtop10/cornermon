# 기기 등록 코드 해시화 대응 (프론트) — Plan

관련 이슈: [#109](https://github.com/lsjtop10/cornermon/issues/109) (프론트 작업 범위), 선행 이슈 [#107](https://github.com/lsjtop10/cornermon/issues/107)(설계 결정), [#108](https://github.com/lsjtop10/cornermon/issues/108)(백엔드, 이미 병합됨 — `7c884f7`)

## 배경

`#107` 논의 결과 확정된 설계:
- 진행자는 `campId` 원문을 알 수 없다. 대신 관리자가 캠프 생성 시 결정적으로 계산되는
  8자 해시 코드(`registrationCode`, Crockford Base32)를 진행자에게 구두/메신저로 공유한다.
- 서버는 `registrationCode` → `campId`를 O(1) 역참조한다. PENDING → 관리자 수동 승인
  게이트는 그대로 유지(B-1 결정).

백엔드는 이미 이 계약으로 `api/swagger.yaml`을 갱신하고 병합했다(`7c884f7`,
`POST /device-registrations` 요청 스키마가 `campId` → `registrationCode` +
`deviceModel` + `displayName`으로 바뀌고, `CampResponse`에 `registrationCode`가
노출됨). **`frontend/lib/shared/api/gen`은 아직 이 갱신 전 스키마를 그대로 갖고 있다** —
`DeviceRegistrationRequest.campId` 필드가 여전히 존재하고 `registrationCode`가 없다
(확인 완료, 아래 Phase A).

## 코드 이해 결과 (선행 조사)

- `frontend/lib/facilitator/session/device_trust_provider.dart`: `requestRegistration`이
  `..campId = registrationCode`로 임시 대응 중(`#107` 문제 3 그대로). stale TODO
  (PENDING→APPROVED 전이 감지 API 없음)도 남아있으나, `GET /device-registrations/me`가
  이미 계약에 존재하는 것으로 확인됨(→ TODO satisfied by existing polling, 본문에서 제거).
- `frontend/lib/facilitator/features/device_pending/device_pending_screen.dart`:
  `_RegistrationForm`이 코드 입력 필드 하나만 갖고 있다. `displayName`(표시용 이름) 입력
  UI가 없다.
- `frontend/lib/admin/features/device_manage/device_manage_screen.dart`: 캠프 스코프
  라우트(`admin_router.dart`에서 `selectedCampIdProvider != null`일 때만 진입 가능)라
  `selectedCampProvider`로 현재 캠프를 이미 구독할 수 있는 위치. 등록 코드 표시에
  적합.
- `frontend/lib/shared/api/domain_aliases.dart`의 `DeviceRegistrationRequestBody`
  typedef는 `DeviceRegistrationRequest`를 그대로 가리키므로, gen 재생성 후 필드만
  바뀌면 이 typedef 자체는 손댈 필요 없음.
- 기기 모델명(`deviceModel`)을 가져올 패키지가 프로젝트에 없음(`Platform.operatingSystem`
  정도만 사용 중). 사용자 확인 결과: 마케팅 모델명이 아니라 **사용자가 기기 설정에서
  지정한 이름**(예: "민수의 iPad")을 우선 사용하기로 함 — `device_info_plus`의
  iOS `name` 필드가 정확히 이 값이다. Android는 OS 차원에서 이런 사용자 지정 이름을
  노출하지 않으므로 `model`(마케팅명, 예: "Pixel 7")로 폴백한다.
- 디자인 톤앤매너는 `frontend/docs/design-system.md` 산문이 아니라 실제 목업
  `관리자 앱 화면 리뷰 — 코너학습_files/saved_resource.html`을 기준으로 맞춘다(기존
  메모 확인). 이 목업에는 등록 코드 표시 화면이 없지만, `.badge`/`.card`/`t-caption`
  토큰과 배지 ID를 monospace로 강조하는 패턴(`#badgeTableBody`)을 그대로 재사용한다.

## Phase B: 진행자 앱 — 등록 요청 페이로드 (예상 소요: 1시간)

### B-1. `device_trust_provider.dart` — 요청 페이로드 갱신 (기존 파일 수정)

```dart
/// 책임: 등록 코드 + 기종 + 표시 이름으로 POST /device-registrations 호출.
/// TODO 주석 제거(GET /device-registrations/me로 폴링 가능함이 확인됨, #107 참고).
Future<void> requestRegistration(
  String registrationCode, {
  required String displayName,
}) async {
  final deviceModel = await _resolveDeviceModel();
  final response = await api.deviceRegistrationsPost(
    request: DeviceRegistrationsPostRequest(
      (DeviceRegistrationsPostRequestBuilder b) => b
        ..registrationCode = registrationCode
        ..deviceName = _defaultDeviceName()
        ..deviceModel = deviceModel
        ..displayName = displayName,
    ),
  );
  // ... 기존 응답 처리 로직 동일
}

/// 책임: iOS는 사용자가 설정에서 지정한 기기 이름을, Android는 마케팅 모델명을 반환.
static Future<String> _resolveDeviceModel() async { ... }
```

- `_resolveDeviceModel()`은 `device_info_plus`의 `DeviceInfoPlugin`을 사용:
  - iOS: `IosDeviceInfo.name` (예: "민수의 iPad")
  - Android: `AndroidDeviceInfo.model` (예: "Pixel 7")
- 실패 시(플러그인 예외) `_defaultDeviceName()`과 동일한 폴백 문자열 사용 — 등록
  요청 자체가 막히면 안 되므로 `deviceModel`은 참고용 필드로 취급.
- `pubspec.yaml`에 `device_info_plus` 의존성 추가.

### B-2. `device_pending_screen.dart` — `displayName` 입력 필드 추가 (기존 파일 수정)

`_RegistrationFormState`에 두 번째 `TextEditingController`(`_displayNameController`) 추가.
`_submit()`이 `ref.read(deviceTrustProvider.notifier).requestRegistration(code, displayName: displayName)` 호출.
"등록 코드" 입력 필드 아래에 "표시용 이름" 필드 배치(레이블: "관리자 화면에 표시될 이름",
힌트: "예: 1번 태블릿"). `canSubmit` 조건에 `_displayNameController.text.trim().isNotEmpty` 추가.

### B-3. `device_pending_screen.dart` — 입력 힌트 문구 보완 (UC-4)

기존 "관리자에게 받은 등록 코드를 입력하세요" 문구는 유지하되, 등록 코드 입력 필드
`hintText`에 형식 힌트 추가(예: `"예: 7ZQK3M2X"`, `api/swagger.yaml`의
`registrationCode` 예시값과 동일). 8자 Crockford Base32라는 내부 구현 디테일은
문구에 노출하지 않는다(UX 관점에서 불필요).

## Phase C: 관리자 앱 — 캠프 등록 코드 노출 (예상 소요: 1시간)

### C-1. `device_manage_screen.dart` — 등록 코드 카드 추가 (기존 파일 수정)

```dart
/// 책임: 현재 선택된 캠프의 registrationCode를 표시하고 탭하면 클립보드로 복사.
class _RegistrationCodeCard extends ConsumerWidget {
  const _RegistrationCodeCard();

  @override
  Widget build(BuildContext context, WidgetRef ref) { ... }
}
```

- `ref.watch(selectedCampProvider)`로 현재 캠프를 구독(이미 캠프 스코프 라우트이므로
  `null` 방어만 하면 됨 — 로딩/에러는 짧게 스피너/숨김 처리).
- `camp.registrationCode`를 monospace 스타일로 크게 표시 + "복사" 아이콘 버튼.
  탭 시 `Clipboard.setData(ClipboardData(text: code))` 후
  `ScaffoldMessenger.of(context).showSnackBar(...)`로 "복사되었습니다" 피드백
  (기존 `track_bulk_manage_screen.dart` 등의 `SnackBar` 확인 패턴과 동일하게 통일).
- 배치 위치: `Scaffold.appBar`의 `bottom`(`TabBar`) 위에 두면 `AppBar`의
  `PreferredSize` 제약과 부딪히므로, `body`의 `TabBarView` 바깥쪽 상단에
  `Column`으로 감싸 카드 → `TabBarView` 순서로 배치(현재 `Scaffold.body`가
  `registrations.when(...)`을 직접 반환하는 구조를 `Column`으로 감싸도록 변경).
- 스타일: 기존 목업(`saved_resource.html`)의 `.badge`류 톤(둥근 배경 pill)이 아니라,
  8자 코드를 강조해야 하므로 카드 배경(`AppColors.bgSurface`) + 큰 monospace
  타이포(`AppTypography.title2`, `fontFamily: monospace` 오버라이드) + 보더
  (`AppColors.border`)로 구성 — `design-system.md`의 `card` 토큰과 동일 반경/패딩
  사용.

### C-2. `camp_ext.dart` — 확장 불필요 확인

`registrationCode`는 단순 필드 노출이라 파생 로직이 없다. `AdminCampX`에 게터를 추가할
필요 없음(YAGNI) — `camp.registrationCode`를 직접 참조.
