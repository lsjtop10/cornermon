# 코너·트랙 관리 화면 계층 구조 전환 + 코너 삭제 UI

## 배경

관리자 대시보드에서 같은 코너가 2개씩 보이는 버그를 조사한 결과, 코너 생성 API가
중복 제출되어 트랙이 하나도 없는 "좀비 코너"가 `corners` 테이블에 실제로 쌓이는 것이
원인이었다. 좀비 코너는 트랙 기준으로 목록을 구성하는 `/corner-track-manage`
화면에는 나타나지 않아 관리자가 존재 자체를 인지하지 못했고, 삭제할 UI 진입점도
없었다(`deleteCornerProvider`는 정의만 있고 어떤 화면도 참조하지 않음).

백엔드는 별도 Issue로 다음을 요청한다 (이 Plan의 스코프 아님):
- `DELETE /corners/{id}`를 hard delete → soft delete로 변경 (요청/응답 계약은 동일하게 유지)
- 목록 조회(`ListCornerViewsByCamp`)에서 soft-delete된 코너 제외
- 트랙 이력이 전혀 없는 soft-delete 코너를 일정 기간 후 물리 삭제하는 배치/스크립트

프론트엔드는 위 계약(요청/응답 모양 불변)을 가정하고, 기존 `DELETE /corners/{id}`
API를 그대로 호출하는 UI만 구현한다. 백엔드가 hard→soft로 구현을 바꿔도 프론트
코드 변경은 필요 없다.

## 개정 (1차 PR 이후)

`backend/db/schema.sql`을 확인한 결과 `tracks.corner_id`, `visits.corner_id`,
`visits.track_id`가 모두 `ON DELETE CASCADE`로 걸려 있어, 코너 삭제 시 트랙과
방문 기록까지 DB가 자동으로 함께 제거한다는 사실을 확인했다. 실제 운영 흐름상
코너를 만들고 트랙을 붙이지 않는 경우는 드물어서, "트랙 0개 코너만 삭제 가능"
조건은 실용성이 낮다고 판단해 다음과 같이 스코프를 넓힌다:

- **삭제 가능 조건 변경**: `tracks.isEmpty` → `tracks.every(status != BUSY)`.
  진행 중인 방문(BUSY 트랙)이 있을 때만 막는다 — CASCADE로 지워지는 트랙이
  실시간으로 스캔 중인 세션이면 현장 진행자의 작업이 끊기기 때문이다. 트랙이
  있어도 전부 IDLE이면 삭제 허용.
- **백엔드 Issue #154(soft-delete 전환)가 아직 미착수**이므로, 이 상태로 병합되면
  당분간 트랙이 있는 코너를 삭제할 때 실제로 hard delete + CASCADE가 그대로
  발생한다(방문 이력 포함 물리 삭제). 백엔드 담당자가 Issue를 곧 수용할 예정이라는
  전제 하에 진행하기로 함 — 릴리스 전에 Issue #154 상태를 다시 확인할 것.
- **UI**: 코너 그룹을 `Card` 각각으로 분리하지 않고, 하나의 `ListView` 안에서
  `ExpansionTile`로 접었다 펼 수 있는 트리뷰 형태로 변경.

## 기존 코드 이해

- `frontend/lib/admin/features/track_bulk_manage/track_bulk_manage_screen.dart`: 현재 `trackListProvider`/`cornerListProvider`를 각각 watch해서 트랙을 플랫 `DataTable`로 그림. 코너 이름은 `cornerNames` 맵으로만 참조(179-201행 정렬 로직 등).
- `frontend/lib/shared/api/providers/corner_track_providers.dart:90-94`: `deleteCornerProvider`(`Future<void> deleteCorner(Ref ref, CornerId id)`) 존재하나 `retry: noRetry`가 없음 — 개발자가이드 §2.3 규칙(부작용 있는 액션 provider는 `retry: noRetry` 명시)을 위반하고 있어 이번 작업 범위에서 함께 수정한다.
- 기존 트랙 삭제(`_delete`, 99-131행) 패턴: BUSY 트랙 포함 시 차단 → 확인 다이얼로그 → provider 호출 → `setState`/`ref.invalidate`. 코너 삭제도 동일 패턴을 따른다.
- `AppButton`은 `size`/`width` 필수 파라미터 — 관리자 화면이므로 `AppButtonSize.compact` 사용(개발자가이드 §5).

## 객체 설계

```dart
/// 코너 1개 + 그 코너에 속한 트랙 목록. 화면 그룹핑 전용, provider 레이어에는 두지 않음.
class CornerTrackGroup {
  const CornerTrackGroup({required this.corner, required this.tracks});
  final api.Corner corner;
  final List<api.Track> tracks;

  /// 트랙·방문 기록은 DB CASCADE로 함께 제거되므로 트랙이 있어도 막지 않는다.
  /// 진행 중인 방문(BUSY 트랙)이 있을 때만 막는다.
  bool get canDelete => tracks.every((t) => t.operationalStatus != api.TrackOperationalStatus.BUSY);
}

/// 코너 목록 + 트랙 목록을 코너 기준으로 묶는다. 트랙이 없는 코너도 그룹으로 포함되어야
/// 좀비 코너가 화면에 드러난다 (현재 버그의 핵심 — 반드시 트랙 유무와 무관하게 전체 코너를 순회).
List<CornerTrackGroup> groupTracksByCorner(
  List<api.Corner> corners,
  List<api.Track> tracks,
);
```

`_TrackBulkManageScreenState`에 추가할 메서드:

```dart
Future<void> _deleteCorner(CampId campId, api.Corner corner) async;
```
